## 0. 이번 리뷰의 핵심 주제

이번 리뷰는 **"공통 모듈을 만들어 놓고 왜 안 쓸까?"**에 초점을 맞추고 있어요.

프로젝트에는 이미 잘 설계된 공통 모듈이 여러 개 있어요.

- `src/api/client.js` — 공통 API 호출 함수 (`request`, `authRequest`)
- `src/api/auth.js` — 인증 관련 API 함수 (`checkWish`, `getProfile`, `toggleWishList`)
- `src/utils/auth.js` — 토큰 유틸리티 (`getToken`, `setToken`, `removeToken`)

그런데 여러 파일에서 이 모듈들을 사용하지 않고 **직접 `fetch()`를 호출하거나 같은 기능을 다시 구현**하고 있어요. 이렇게 되면 코드가 중복되고, 수정할 때 여러 곳을 동시에 바꿔야 해서 유지보수가 어려워져요.

이번 리뷰에서는 이런 **결합도(Coupling)와 응집도(Cohesion)** 관점의 이슈를 중심으로 살펴볼게요.

## 1. API 호출 일관성 문제 — client.js를 만들어 놓고 직접 fetch를 호출하고 있어요 (중요도: 높음)

**관련 파일:** `wishlist/index.js`, `reservations-check/index.js`, `reservations-detail/index.js`, `profile/profile.js`, `reservation-request/reservation-request.js`

`src/api/client.js`에 공통 `request()` 함수가 이미 잘 만들어져 있어요. 토큰 처리, 헤더 설정, 에러 처리까지 모두 포함되어 있어요.

```js
// src/api/client.js — 이미 잘 만들어진 공통 함수!
export async function request(endpoint, options = {}) {
  const token = getToken();
  const headers = {
    'Content-Type': 'application/json',
    ...options.headers,
  };
  if (token) {
    headers.Authorization = `Bearer ${token}`;
  }
  const res = await fetch(`${BASE_URL}${endpoint}`, { ...options, headers });
  if (!res.ok) {
    throw new Error(`HTTP 에러: ${res.status}`);
  }
  return res.json();
}
```

그리고 `src/api/auth.js`에서는 이 `request()`를 잘 사용하고 있어요.

```js
// src/api/auth.js — 올바른 패턴! client.js의 request()를 사용
export async function checkWish(accommodationIds) {
  return authRequest('/me/wishlist/check', {
    method: 'POST',
    body: JSON.stringify({ accommodationIds }),
  });
}
```

그런데 아래 파일들에서는 이 공통 함수를 사용하지 않고 **직접 fetch를 호출**하고 있어요.

```js

  headers: {
    Authorization: `Bearer ${token}`,
```

**왜 문제일까요?**

직접 `fetch()`를 호출하면 **결합도가 높아져요**. 나중에 API 서버 URL이 바뀌거나, 인증 헤더 형식이 변경되거나, 에러 처리 방식을 통일하고 싶을 때 **14곳 이상을 모두 찾아서 수정**해야 해요. `client.js`의 `request()`를 사용하면 **한 곳만 수정**하면 돼요.

특히 `wishlist/index.js`의 `checkWished()` 함수는 `src/api/auth.js`에 이미 똑같은 기능의 `checkWish()` 함수가 있는데도 직접 구현하고 있어요. 같은 프로젝트의 `src/landing.js`에서는 `checkWish`를 auth.js에서 import해서 사용하고 있으니, 이 패턴을 따르면 돼요.

**추천하는 방법.**

필요한 API 함수를 `src/api/` 디렉토리에 추가하고, 각 페이지에서 import해서 사용하세요.

```js
// src/api/reservation.js (신규) — 예약 관련 API를 한 곳에 모아요
import { request, authRequest } from './client.js';

export async function getReservations() {
  return authRequest('/me/reservations');
}

export async function getReservationDetail(reservationId) {
  return authRequest(`/reservations/${reservationId}`);
}

export async function cancelReservation(reservationId) {
  return authRequest(`/reservations/${reservationId}/cancel`, {
    method: 'PATCH',
  });
}

export async function createReservation(data) {
  return authRequest('/reservations', {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

```js
// reservations-check/index.js — client.js를 활용하면 이렇게 간결해져요!
import { getReservations } from '../src/api/reservation.js';

const data = await getReservations();
```

## 2. 인증 관련 코드 중복 — 같은 기능을 파일마다 다시 만들고 있어요 (중요도: 높음)

**관련 파일:** `reservations-detail/index.js`, `reservation-request/reservation-request.js`, `profile/profile.js`, `wishlist/index.js`

`src/utils/auth.js`에 `getToken`, `setToken`, `removeToken`이 이미 있어요. 그런데 `reservations-detail/index.js`에서는 이 유틸리티를 import하지 않고 **자체 함수를 다시 만들어** 사용하고 있어요.

```js
// reservations-detail/index.js 13~23줄 — auth.js와 동일한 기능을 직접 구현!
function getToken() {
  return localStorage.getItem(storageKeys.token);
}

function getReservationId() {
  return localStorage.getItem(storageKeys.reservationId);
}

function clearReservationId() {
  localStorage.removeItem(storageKeys.reservationId);
}
```

토큰 검증 로직도 파일마다 직접 구현하고 있어요.


```js

async function fetchProfile() {
  if (!token) {
    alert('로그인을 먼저 해주세요');

```

```js

if (token) {
```

**왜 문제일까요?**

"로그인이 필요합니다"라는 **동일한 기능**이 4곳에서 각각 다른 방식으로 구현되어 있어요. **응집도가 낮은 전형적인 사례**에요. 나중에 "로그인 페이지 경로가 바뀌었다"거나 "401 에러 처리 방식을 바꾸고 싶다"면 4곳을 모두 찾아서 수정해야 해요.

이전 리뷰(3-1)에서 이미 `src/auth.js`로 분리하는 것을 제안했었어요. 해당 제안을 참고하여 인증 체크 로직을 한 곳으로 모아주세요.

## 3. 에러 메시지 요소 네이밍이 의미를 전달하지 못해요 (중요도: 보통)

**관련 파일:** `login/index.html`, `signup/index.html`

HTML에서 에러 메시지를 표시하는 요소의 id가 `result1`, `result2`, `result3`처럼 **숫자로만** 구분되어 있어요.

```html
<p id="result1" class="text-xs text-negative-500"></p>   <!-- 아이디 에러 -->
<p id="result2" class="text-xs text-negative-500"></p>   <!-- 비밀번호 에러 -->
<p id="result3" class="text-xs text-negative-500 text-center"></p>  <!-- 공통 에러 -->
```

```html
<!-- signup/index.html -->
<p id="result1" class="text-xs text-negative-500"></p>   <!-- 아이디 에러 -->
<p id="result2" class="text-xs text-negative-500"></p>   <!-- 비밀번호 에러 -->
<p id="result3" class="text-xs text-negative-500"></p>   <!-- 이름 에러 -->
<p id="result4" class="text-xs text-negative-500"></p>   <!-- 전화번호 에러 -->
<p id="result5" class="text-xs text-negative-500 text-center"></p>  <!-- 공통 에러 -->
```

JS에서도 이 숫자 id를 그대로 쓰고 있어서 어떤 필드의 에러인지 코드만 보고는 알 수 없어요.

```js
// login/index.js 10~14줄
const errorMessage = {
  id: document.getElementById('result1'),   // result1이 뭘까?
  pw: document.getElementById('result2'),   // result2가 뭘까?
  common: document.getElementById('result3'),
};
```

**왜 문제일까요?**

코드를 처음 보는 사람이 `result1`이 어떤 에러 메시지인지 알 수 없어요. HTML 파일을 열어서 확인해야 해요. id 이름만 봐도 용도를 알 수 있으면 코드 가독성이 훨씬 좋아져요.

**추천하는 방법.**

```html

<p id="error-username" class="text-xs text-negative-500"></p>
<p id="error-password" class="text-xs text-negative-500"></p>

```

```js
  id: document.getElementById('error-username'),
  pw: document.getElementById('error-password'),
  common: document.getElementById('error-common'),
};
```

## 4. 비밀번호 토글이 mouseover/mouseout으로 동작해요 (중요도: 보통)

**관련 파일:** `login/index.js`, `signup/index.js`

비밀번호 보기/숨기기 버튼이 마우스를 올려놓는 동안만 보이는 방식이에요.

```js
// login/index.js 24~29줄
togglePass.addEventListener('mouseover', () => {
  formElements.pw.type = 'text';
});
togglePass.addEventListener('mouseout', () => {
  formElements.pw.type = 'password';
});
```

```js
// signup/index.js 35~40줄 — 동일한 코드
togglePass.addEventListener('mouseover', () => {
  formElements.pw.type = 'text';
});
togglePass.addEventListener('mouseout', () => {
  formElements.pw.type = 'password';
});
```

**왜 문제일까요?**

두 가지 문제가 있어요.

첫째, **모바일 접근성 문제**에요. 스마트폰에는 마우스가 없기 때문에 `mouseover` 이벤트가 발생하지 않아요. 모바일 사용자는 비밀번호를 확인할 방법이 없어요.

둘째, **UX 관점**에서 대부분의 웹사이트는 비밀번호 토글을 **클릭** 방식으로 구현하고 있어요. 한 번 누르면 보이고, 다시 누르면 숨겨지는 방식이 사용자에게 더 익숙해요.

**추천하는 방법.**

```js
// click 토글 방식
togglePass.addEventListener('click', () => {
  const isPassword = formElements.pw.type === 'password';
  formElements.pw.type = isPassword ? 'text' : 'password';
});
```

## 5. console.log로 사용자 비밀번호가 콘솔에 출력돼요 (중요도: 높음)

**관련 파일:** `signup/index.js`, `wishlist/index.js`

회원가입 시 사용자가 입력한 데이터가 **비밀번호를 포함하여** 콘솔에 출력돼요.

```js
// signup/index.js 152줄 — 비밀번호가 포함된 객체를 콘솔에 출력!
const signupData = getsignupData();
console.log(signupData);
// 콘솔 출력: { username: "user1", password: "mySecret123!", name: "홍길동", phone: "01012345678" }
```

```js
// wishlist/index.js 25줄
console.log('유효한 토큰입니다.');
```

**왜 문제일까요?**

`signupData`에는 사용자의 **비밀번호**가 포함되어 있어요. 이 코드가 배포되면 누구나 브라우저 개발자 도구(F12)를 열어서 다른 사용자가 입력한 비밀번호를 볼 수 있어요. 이것은 **보안 취약점**이에요.

**추천하는 방법.**

디버깅이 끝났으면 `console.log(signupData)` 줄을 삭제하세요. `wishlist/index.js`의 `console.log('유효한 토큰입니다.')`도 삭제하세요. 정상 동작을 확인하는 용도의 `console.log`는 개발이 끝나면 제거하는 것이 좋아요.

## 6. 인라인 스타일 사용 — TailwindCSS 클래스를 사용해야 해요 (중요도: 보통)

**관련 파일:** `reservations-check/index.js`, `reservation-request/reservation-request.js`

Tailwind CSS를 사용하는 프로젝트인데, JavaScript에서 `style` 속성을 직접 변경하고 있어요.

```js
// reservations-check/index.js 16줄
elements.result.style.color = 'red';
```

```js
// reservation-request/reservation-request.js 200~201줄
calModal.style.display = 'flex';
document.body.style.overflow = 'hidden';
```

**왜 문제일까요?**

Tailwind CSS를 쓰기로 한 프로젝트에서 `style.color = 'red'`나 `style.display = 'flex'`를 쓰면 스타일 관리 포인트가 분산돼요. HTML의 Tailwind 클래스를 보고 "이 요소의 스타일은 이렇구나"라고 생각했는데, 실제로는 JS에서 인라인 스타일로 덮어쓰고 있으면 혼란스러워요.

**추천하는 방법.**

```js
// reservations-check/index.js — Tailwind 클래스로 대체
elements.result.classList.add('text-negative-500');     // style.color = 'red' 대신
elements.result.classList.remove('text-negative-500');  // style.color = '' 대신
```

```js
// reservation-request/reservation-request.js — 클래스 토글 방식
calModal.classList.add('active');     // style.display = 'flex' 대신
calModal.classList.remove('active'); // style.display = '' 대신
```

이전 리뷰(3-5)에서 제안한 모달 패턴 통일과 함께 수정하면 좋아요.

## 7. 변수명 카멜케이스가 지켜지지 않는 곳이 있어요 (중요도: 보통)

**관련 파일:** `login/index.js`, `signup/index.js`

```js
// login/index.js 55줄
function getloginData() {   // ← 'l'이 소문자
```

```js
// signup/index.js 71줄
function getsignupData() {  // ← 's'가 소문자
```

**왜 문제일까요?**

JavaScript에서는 함수명과 변수명에 **카멜케이스(camelCase)**를 사용하는 것이 표준 컨벤션이에요. 두 번째 단어의 첫 글자는 대문자로 시작해야 해요. 프로젝트의 다른 코드에서는 `clearMessages()`, `showError()`, `formatMonthDay()` 등 카멜케이스를 잘 지키고 있어요.

**추천하는 방법.**

```js
function getLoginData() {   // ← 'L' 대문자
function getSignupData() {  // ← 'S' 대문자
```

호출하는 곳도 함께 변경해야 해요.

```js
// login/index.js 91줄
const loginData = getLoginData();    // getloginData → getLoginData

// signup/index.js 151줄
const signupData = getSignupData();  // getsignupData → getSignupData
```

**왜 문제일까요?**

서버에서 받아온 데이터(숙소 이름, 위치, 이미지 URL 등)를 `innerHTML`의 템플릿 리터럴에 직접 넣으면, 악성 데이터가 HTML로 해석돼요. 예를 들어 숙소 이름이 `<img src=x onerror=alert(document.cookie)>`라면 사용자의 쿠키가 탈취될 수 있어요. 이것을 **XSS(크로스 사이트 스크립팅)** 공격이라고 해요.

**추천하는 방법.**

이전 리뷰(3-3)에서 제안한 것처럼, HTML 구조는 `innerHTML`로 만들되 **데이터는 `textContent`와 속성으로 삽입**하세요.

```js
// 안전한 방법 예시
item.innerHTML = `
  <img class="w-full h-40 object-cover rounded-md" />
  <p class="mt-2 font-bold"></p>
  <p class="text-sm text-gray-500"></p>
`;
item.querySelector('img').src = accommodation.thumbnailUrl;
item.querySelector('img').alt = accommodation.title;
item.querySelector('.font-bold').textContent = accommodation.title;
item.querySelector('.text-gray-500').textContent =
  `${checkIn} ~ ${checkOut} | ${schedule.nights}박`;
```

## 9. Enter 키 처리 로직이 두 곳에서 중복돼요 — form 태그를 활용하면 불필요한 코드에요 (중요도: 보통)

**관련 파일:** `login/index.js`, `signup/index.js`

두 파일에 거의 동일한 `enterPress()` 함수가 있어요.

```js
// login/index.js 35~53줄
function enterPress() {
  const forms = [formElements.id, formElements.pw];
  forms.forEach((form, idx) => {
    form.addEventListener('keydown', (e) => {
      if (e.key !== 'Enter') return;
      e.preventDefault();
      const nextForm = forms[idx + 1];
      if (nextForm) {
        nextForm.focus();
      } else {
        formElements.loginbtn.click();
      }
    });
  });
}
```

```js
// signup/index.js 46~69줄 — 거의 동일한 코드!
function enterPress() {
  const forms = [formElements.id, formElements.pw, formElements.nm, formElements.pn];
  forms.forEach((form, idx) => {
    // ... 동일한 로직
  });
}
```

**왜 문제일까요?**

두 가지 문제가 있어요.

첫째, 같은 로직이 두 곳에 **복사-붙여넣기** 되어 있어요. 응집도가 낮아요.

둘째, 사실 이 코드 자체가 **불필요**해요. 두 페이지 모두 이미 `<form>` 태그를 사용하고 있고, `submit` 이벤트를 처리하고 있어요. HTML `<form>` 안에서 Enter 키를 누르면 **자동으로 submit 이벤트가 발생**해요. `e.preventDefault()`로 기본 동작을 막고 다시 수동으로 구현할 필요가 없어요.

**추천하는 방법.**

`enterPress()` 함수와 호출 부분을 모두 삭제하세요. `<form>` 태그의 기본 동작이 알아서 처리해줘요. 마지막 입력 필드에서 Enter를 누르면 자동으로 submit 이벤트가 발생하고, 이미 등록된 `signupForm.addEventListener('submit', ...)` 핸들러가 동작해요.

## 10. wishlist/index.js에서 auth.js에 이미 있는 함수를 직접 구현하고 있어요 (중요도: 보통)

**관련 파일:** `wishlist/index.js`, `src/api/auth.js`, `src/landing.js`

`src/api/auth.js`에는 이미 `checkWish()` 함수가 있어요.

```js
// src/api/auth.js 24~29줄 — 이미 존재하는 함수!
export async function checkWish(accommodationIds) {
  return authRequest('/me/wishlist/check', {
    method: 'POST',
    body: JSON.stringify({ accommodationIds }),
  });
}
```

`src/landing.js`에서는 이 함수를 잘 import해서 사용하고 있어요. 그런데 `wishlist/index.js`에서는 같은 기능을 14줄이나 되는 코드로 직접 구현하고 있어요.

```js
// wishlist/index.js 60~82줄 — checkWish와 동일한 기능을 직접 구현!
async function checkWished() {
  const accommodationIds = [...roomData.keys()];
  const res = await fetch(`${constants.API_BASE_URL}/me/wishlist/check`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      Authorization: `Bearer ${token}`,
    },
    body: JSON.stringify({ accommodationIds: accommodationIds }),
  });
  const { success, message, data } = await res.json();
  if (!success) {
    console.log('찜 일괄 체크 실패');
    console.log(message);
    return;
  }
  for (let id of data.wishlistedAccommodationIds) {
    roomData.get(id).setWish(true);
  }
}
```

**왜 문제일까요?**

같은 프로젝트 안에서 같은 기능을 하는 코드가 두 가지 방식으로 존재하면 혼란스러워요. `landing.js`를 보면 "checkWish는 auth.js에서 가져오면 되는구나"라고 이해하지만, `wishlist/index.js`를 보면 "여기서는 직접 구현하네?"라고 헷갈려요.

**추천하는 방법.**

```js
// wishlist/index.js — auth.js의 checkWish를 가져다 쓰세요!
import { checkWish } from '../src/api/auth.js';

async function checkWished() {
  const accommodationIds = [...roomData.keys()];
  const result = await checkWish(accommodationIds);
  if (!result || !result.success) return;

  for (let id of result.data.wishlistedAccommodationIds) {
    roomData.get(id).setWish(true);
  }
}
```

14줄이 5줄로 줄어들어요.

## 12. emptyState 컴포넌트의 비효율적 패턴 (중요도: 권장)

**관련 파일:** `src/components/emptyState.js`

```js
// src/components/emptyState.js
export function buildEmptyState(message) {
  const div = document.createElement('div');
  div.className = 'w-full min-h-140 col-span-3 ...';
  div.innerHTML = `${message} <span></span> 비어있습니다.`;

  const highlight = div.querySelector('span');  // innerHTML로 넣은 걸 다시 querySelector로 찾음
  highlight.className = '...';
  highlight.textContent = '텅...';

  return div;
}
```

**왜 문제일까요?**

`innerHTML`로 HTML 문자열을 넣은 뒤 바로 `querySelector`로 찾아서 속성을 설정하고 있어요. 이전 리뷰(3-2)에서 제안한 방식대로 `createElement`와 `append`를 사용하면 더 깔끔하고, XSS 위험도 없어요.

**추천하는 방법.**

```js
export function buildEmptyState(message) {
  const div = document.createElement('div');
  div.className = 'w-full min-h-140 col-span-3 flex gap-20 items-center justify-center text-lg font-semibold text-shark-600';

  const text = document.createTextNode(message + ' ');
  const highlight = document.createElement('span');
  highlight.className = 'inline-block text-[10rem] font-extrabold text-primary-500 align-text-top';
  highlight.textContent = '텅...';
  const suffix = document.createTextNode(' 비어있습니다.');

  div.append(text, highlight, suffix);
  return div;
}
```

## 마무리

이번 리뷰의 핵심 메시지는 **"이미 만든 공통 모듈을 적극적으로 활용하자"**에요.

`src/api/client.js`의 `request()` 함수, `src/api/auth.js`의 API 함수들, `src/utils/auth.js`의 토큰 유틸리티는 모두 잘 설계되어 있어요. 문제는 이것들을 **일부 파일에서만 사용**하고, 나머지 파일에서는 같은 기능을 직접 구현하고 있다는 점이에요.

공통 모듈을 활용하면 세 가지 이점이 있어요.

첫째, **코드량이 줄어요**. wishlist의 `checkWished()` 함수가 14줄에서 5줄로 줄어드는 것처럼, 중복 코드가 사라져요.

둘째, **수정이 쉬워요**. API URL이 바뀌거나 인증 방식이 변경되면 `client.js` 한 곳만 수정하면 전체 프로젝트에 반영돼요.

셋째, **버그가 줄어요**. 같은 기능을 여러 곳에서 구현하면 어떤 곳은 에러 처리를 빠뜨리거나, 어떤 곳은 토큰을 다르게 처리하는 등 미묘한 차이가 생겨요.

`src/api/auth.js`가 `client.js`의 `request()`를 사용하는 방식이 이 프로젝트의 **올바른 패턴**이에요. 다른 파일들도 이 패턴을 따라주세요. 화이팅!
