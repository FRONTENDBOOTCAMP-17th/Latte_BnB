# Latte BnB 프로토타입 구현 과정 문서

## 1. 문서 개요

본 문서는 **Latte BnB** 프로젝트의 프로토타입 화면 및 사용자 흐름을 구상하는 과정에서 정리된 메모를 바탕으로, 서비스의 핵심 기능을 중심으로 한 **프로토타입 구현 방향**, **화면별 역할**, **정보 구조**, **접근 권한**, **주요 인터랙션**, **우선순위 판단 근거**를 전문적인 문서 형식으로 재정리한 것이다.

이 문서의 목적은 다음과 같다.

* 제한된 프로젝트 기간 내에 핵심 기능 중심의 프로토타입 범위를 명확히 정의한다.
* 화면별로 어떤 기능과 데이터를 우선 구현해야 하는지 정리한다.
* 사용자 흐름을 기준으로 페이지 간 연결 관계를 명확히 한다.
* 실제 개발 이전 단계에서 기획, UI 설계, API 설계의 기준 문서로 활용한다.

## 2. 프로젝트 방향성

### 2.1 핵심 목표

Latte BnB는 숙소 탐색부터 예약까지의 기본 흐름을 구현하는 것을 목표로 하는 **숙소 예약 서비스 클론 프로젝트**이다. 프로토타입 단계에서는 실제 상용 서비스 수준의 모든 기능을 구현하기보다, 아래의 핵심 사용자 경험을 완성하는 데 집중한다.

1. 사용자가 숙소 목록을 탐색할 수 있다.
2. 사용자가 숙소를 검색할 수 있다.
3. 사용자가 회원가입 및 로그인을 할 수 있다.
4. 사용자가 숙소를 찜할 수 있다.
5. 사용자가 숙소 상세를 확인하고 예약을 진행할 수 있다.
6. 사용자가 자신의 예약 내역과 프로필 정보를 확인할 수 있다.

### 2.2 구현 우선순위 원칙

프로토타입은 약 20일 미만의 짧은 기간 안에 완성해야 하므로, 다음 원칙을 따른다.

| 원칙              | 설명                                                   |
| --------------- | ---------------------------------------------------- |
| 핵심 기능 우선        | 숙소 탐색, 상세 조회, 예약, 찜, 로그인 등 서비스 본질에 해당하는 기능을 우선 구현한다. |
| 부가기능 축소         | 테마별 인기 숙소, 과도한 프로필 커스터마이징, 소셜 로그인 등은 제외하거나 후순위로 둔다.  |
| 단순하고 안정적인 구조 선택 | 구현 난도가 높고 최적화 부담이 큰 구조보다, 일정 내 구현 가능한 구조를 채택한다.      |
| 일관된 UX 유지       | 모바일, 태블릿, 데스크탑에서 공통된 정보 구조를 유지하고 배치만 반응형으로 조정한다.     |

## 3. 프로토타입 범위 정의

### 3.1 포함 범위

| 영역     | 포함 내용                            |
| ------ | -------------------------------- |
| 랜딩 페이지 | 전체 숙소 리스트 조회, 검색, 정렬, 페이지네이션     |
| 인증     | 회원가입, 로그인, 로그아웃                  |
| 위시리스트  | 찜 목록 조회, 찜/찜 해제                  |
| 숙소 상세  | 숙소 정보 조회, 예약 요청 페이지 진입           |
| 예약 요청  | 예약 정보 입력, 예약 확정                  |
| 내 예약   | 예약 목록 조회, 예약 상세 조회, 예약 취소        |
| 프로필    | 내 정보 조회, 로그아웃, 회원 탈퇴, 정책/약관/회사정보 |

### 3.2 제외 또는 후순위 범위

| 영역                | 제외/후순위 이유                          |
| ----------------- | ---------------------------------- |
| 테마별 인기 숙소 리스트     | 핵심 탐색 흐름이 아닌 부가 기능으로 판단            |
| 무한 스크롤            | 구현 및 최적화 비용이 프로젝트 기간 대비 부담됨        |
| 소셜 로그인            | 외부 인증 연동 범위가 커지고 핵심 기능과 직접 관련성이 낮음 |
| 전화번호 인증 문자 발송     | 외부 메시지 서비스 연동이 필요하여 범위 밖           |
| 예약 draft 생성/수정 흐름 | 프로토타입 범위를 단순화하기 위해 제외              |
| 고도화된 프로필 기능       | 예약 서비스의 핵심 흐름과 거리가 있음              |

## 4. 정보 구조 및 사용자 흐름

### 4.1 전체 사용자 흐름

```md
랜딩 페이지 진입
→ 숙소 목록 탐색
→ 검색 또는 정렬
→ 숙소 상세 페이지 진입
→ 예약하기 클릭
→ 예약 요청 페이지 이동
→ 체크인/체크아웃 날짜 선택
→ 게스트 인원 설정
→ 예약 확정
→ 예약 상세 페이지 이동
```

로그인 관련 흐름은 다음과 같다.

```md
비로그인 상태
→ 랜딩 페이지, 숙소 상세 페이지 접근 가능
→ 찜/예약/내 예약/프로필 접근 시 로그인 필요
→ 회원가입 또는 로그인 수행
→ 로그인 후 보호된 기능 접근 가능
```

### 4.2 예약 흐름 개정 요약

기존에는 숙소 상세 페이지에서 예약 draft를 생성하고, 예약 요청 페이지에서 draft를 수정하는 구조를 고려하였다. 그러나 프로토타입 범위를 단순화하기 위해 해당 흐름은 제외하였다. 이 변경사항은 기존 예약 관련 섹션과 API 정리에도 영향을 준다. 

개정된 예약 흐름은 다음과 같다.

* 숙소 상세 페이지에서는 숙소 정보를 보여주고 **예약 요청 페이지로 이동시키는 역할만 수행**한다.
* 체크인/체크아웃 날짜 선택 기능은 **예약 요청 페이지로 이동**한다.
* 게스트 인원 설정 역시 **예약 요청 페이지에서 수행**한다.
* 예약 요청 페이지에서는 **수정 기능 없이 입력과 확정만 담당**한다.
* 예약 확정 시 결제가 완료되었다고 가정하고, 즉시 **예약 상세 페이지로 이동**한다.

## 5. 랜딩 페이지 프로토타입 구현 방향

### 5.1 랜딩 페이지의 역할

랜딩 페이지는 서비스 진입 시 사용자가 가장 먼저 마주하는 화면이며, 가장 중요한 기능은 **숙소 탐색 시작점**을 제공하는 것이다.

초기 아이디어로는 “테마별 인기 숙소 리스트” 제공도 고려했으나, 프로토타입의 핵심은 사용자가 숙소를 탐색하고 예약으로 이어지는 흐름을 빠르게 경험하는 것이므로, 최종적으로는 **전체 숙소 리스트 중심 구조**가 더 합리적이라고 판단하였다.

### 5.2 리스트 구성 전략

#### 전체 숙소 리스트 채택 이유

| 항목   | 판단                           |
| ---- | ---------------------------- |
| 핵심성  | 전체 숙소 탐색은 서비스의 핵심 기능에 해당     |
| 단순성  | 화면 구조와 데이터 흐름이 명확해짐          |
| 유지보수 | 테마별 섹션보다 구현, 테스트, API 설계가 단순 |
| 확장성  | 이후 인기순, 테마별 분류를 추가하기 쉬움      |

#### 테마별 인기 숙소를 제외한 이유

* 사용자가 실제 예약으로 이어지기 위해 반드시 필요한 정보는 아니다.
* 메인 화면이 과도하게 복잡해질 수 있다.
* 기획, 디자인, API, 데이터 가공 범위가 커진다.
* 프로젝트 기간 내 완성도를 높이기 어렵다.

## 6. 리스트 로딩 방식 결정

### 6.1 무한 스크롤 vs 페이지네이션 검토

| 항목      | 무한 스크롤          | 페이지네이션      |
| ------- | --------------- | ----------- |
| 사용자 경험  | 연속 탐색에 유리       | 페이지 전환이 필요  |
| 구현 난이도  | 상대적으로 높음        | 비교적 단순      |
| 성능 관리   | 가상화 등 최적화 고려 필요 | 상대적으로 안정적   |
| 테스트 편의성 | 복잡함             | 쉬움          |
| 일정 적합성  | 짧은 기간에 부담       | 프로젝트 기간에 적합 |

### 6.2 최종 판단: 페이지네이션

프로토타입 단계에서는 **페이지네이션**을 채택한다.

#### 판단 근거

1. 프로젝트 기간이 20일 미만으로 짧다.
2. 무한 스크롤은 단순히 “계속 불러오는 기능”이 아니라, 긴 스크롤 환경에서 렌더링 성능 문제를 동반한다.
3. 페이지네이션은 구현과 테스트가 단순하여 전체 완성도를 높이기 유리하다.
4. 숙소 탐색 서비스 특성상 페이지 기반 탐색도 충분히 자연스럽다.

## 7. 랜딩 페이지 레이아웃 구조

### 7.1 반응형 구조

#### 모바일 / 태블릿

* 기준 폭: 모바일 최소 400px, 태블릿 최소 640px
* 구조

  * 헤더(검색 바 영역)
  * 숙소 리스트
  * 하단 고정 네비게이션
  * 푸터

#### 데스크탑

* 기준 폭: 1024px 이상
* 구조

  * 헤더(검색 바 + 우측 햄버거 메뉴)
  * 숙소 리스트
  * 푸터

### 7.2 숙소 카드 배치 전략

숙소 카드의 최소 크기를 **360px**으로 설정한다.

이 기준은 다음과 같은 장점을 가진다.

* 카드 내부 정보가 과도하게 압축되지 않는다.
* 썸네일 이미지, 이름, 지역, 가격, 평점 등 핵심 정보를 충분히 배치할 수 있다.
* 모바일 1열, 태블릿 1~2열, 데스크탑 3~4열로 자연스럽게 확장 가능하다.

### 7.3 열 수 설계

| 디바이스 | 예상 열 수 |
| ---- | ------ |
| 모바일  | 1열     |
| 태블릿  | 1~2열   |
| 데스크탑 | 최대 4열  |

1920px 수준의 넓은 화면에서는 이론상 5열도 가능하지만, 프로토타입에서는 콘텐츠 밀도를 지나치게 높이지 않고 **여백의 미**를 유지하기 위해 최대 4열을 상한으로 두는 것이 적절하다고 판단한다.

### 7.4 페이지당 표시 개수

* 한 페이지당 **20개**
* 데스크탑 기준 최대 4열 × 5행 구성 가능

이 방식은 다음과 같은 장점이 있다.

* 페이지 높이가 지나치게 짧지 않아 화면이 풍성해 보인다.
* 한 번에 보여주는 데이터 양이 적절하다.
* 페이지네이션 UI와도 잘 맞는다.

## 8. 숙소 카드 정보 설계

숙소 카드에는 사용자가 상세 페이지 진입 여부를 판단할 수 있는 최소한의 핵심 정보를 보여준다.

| 항목         | 필요성                      |
| ---------- | ------------------------ |
| 대표 썸네일 이미지 | 숙소의 첫인상을 결정하는 핵심 요소      |
| 숙소 이름      | 숙소 식별을 위한 기본 정보          |
| 지역         | 사용자의 위치 기반 탐색 판단에 필요     |
| 숙박 기간      | 검색 조건 또는 노출 조건에 따라 표시 가능 |
| 평점         | 신뢰성과 품질 인식에 도움           |
| 가격         | 예약 의사결정의 핵심 기준           |

### 권장 카드 정보 우선순위

1. 썸네일 이미지
2. 숙소 이름
3. 지역
4. 평점
5. 1박 기준 가격 또는 총 가격
6. 숙박 가능 일정 또는 선택한 일정 정보(조건부)

## 9. 검색 기능 설계 방향

### 9.1 검색의 목적

검색 기능은 사용자가 원하는 숙소를 빠르게 좁혀서 찾도록 돕는 기능이다. 프로토타입 단계에서는 복잡한 검색 필터 시스템보다, **이름과 지역을 기반으로 한 텍스트 검색**을 우선 구현하는 것이 적절하다.

### 9.2 초기 검색 정책

| 검색 대상 | 방식        |
| ----- | --------- |
| 숙소 이름 | 검색어 포함 여부 |
| 지역    | 검색어 포함 여부 |

즉, 사용자가 입력한 검색어가 숙소 이름 또는 지역 문자열에 포함되면 검색 결과에 포함시키는 방식이다.

### 9.3 이 방식의 장점

* 구현이 단순하다.
* 사용자가 기대하는 기본 검색 경험을 제공할 수 있다.
* API 설계가 복잡해지지 않는다.
* 이후 체크인/체크아웃/인원/가격범위 등의 필터를 추가하기 쉽다.

### 9.4 정렬 기능

클라이언트 측 정렬 기능으로 다음을 고려한다.

| 정렬 기준 | 설명                         |
| ----- | -------------------------- |
| 인기순   | 리뷰 수, 평점, 찜 수 등 기준으로 정렬 가능 |
| 가격순   | 가격 오름차순 / 내림차순             |

단, 프로토타입 단계에서는 실제 정렬 기준의 복잡성을 낮추기 위해 단순화된 기준을 사용해도 무방하다.

## 10. 인증 기능 프로토타입 설계

### 10.1 접근 가능한 사용자

| 사용자 상태 | 랜딩 페이지 접근 | 숙소 조회 | 찜  | 예약 | 내 예약 | 프로필 |
| ------ | --------- | ----- | -- | -- | ---- | --- |
| 비로그인   | 가능        | 가능    | 불가 | 불가 | 불가   | 불가  |
| 로그인    | 가능        | 가능    | 가능 | 가능 | 가능   | 가능  |

### 10.2 회원가입 방식 결정

실제 에어비앤비는 다양한 로그인/회원가입 방식을 제공하지만, 본 프로젝트에서는 다음 이유로 일반 회원가입 방식을 채택한다.

* 소셜 로그인 제외
* 전화번호 인증 문자 발송 기능 제외
* 핵심 기능 완성에 집중

따라서 회원가입 정보는 다음과 같이 구성한다.

| 항목   | 설명           |
| ---- | ------------ |
| 아이디  | 정규표현식 검증 적용  |
| 비밀번호 | 정규표현식 검증 적용  |
| 이름   | 기본 사용자 식별 정보 |
| 전화번호 | 기본 연락처 정보    |

### 10.3 로그인/회원가입 흐름

#### 회원가입

1. 사용자가 회원가입 페이지에 진입한다.
2. 아이디, 비밀번호, 이름, 전화번호를 입력한다.
3. 정규표현식으로 아이디와 비밀번호 형식을 검증한다.
4. 회원가입 버튼 클릭 시 가입 요청을 보낸다.
5. 성공 시 로그인 화면 또는 메인 화면으로 이동한다.

#### 로그인

1. 사용자가 아이디와 비밀번호를 입력한다.
2. 로그인 버튼 클릭 시 인증 요청을 보낸다.
3. 성공 시 토큰을 저장한다.
4. 이후 보호된 페이지 접근이 가능하다.
5. 실패 시 “아이디/비밀번호를 다시 입력해주세요”와 같은 에러 메시지를 출력한다.

### 10.4 토큰 저장에 대한 정리

메모에는 “아이디와 비밀번호는 토큰에 저장된다”는 표현이 있으나, 정확한 구현 관점에서는 **아이디/비밀번호 자체를 저장하는 것이 아니라**, 로그인 성공 후 발급된 **인증 토큰(JWT 등)** 을 저장하는 구조가 적절하다.

즉, 문서상 표현은 아래와 같이 정리하는 것이 바람직하다.

> 사용자는 아이디와 비밀번호로 로그인하고, 로그인 성공 시 발급된 인증 토큰을 클라이언트에 저장한다.

## 11. 위시리스트 페이지 구현 방향

### 11.1 위시리스트의 역할

위시리스트는 사용자가 관심 있는 숙소를 저장해두는 개인화 기능이다. 이는 사용자별 데이터이므로 **로그인 상태에서만 접근 가능**해야 한다.

### 11.2 핵심 상호작용

* 숙소 카드의 하트 아이콘을 눌러 찜/찜 해제
* 위시리스트 페이지에서는 찜된 숙소만 조회
* 하트 상태에 따라 시각적으로 구분

### 11.3 카드 상태

| 상태    | UI 표현             |
| ----- | ----------------- |
| 찜 안 됨 | 비어 있는 하트          |
| 찜 됨   | 파스텔톤 붉은색으로 채워진 하트 |

### 11.4 데이터 흐름

1. 사용자가 위시리스트 페이지에 접근한다.
2. 서버에 GET 요청을 보낸다.
3. 찜 상태인 숙소 목록만 응답받는다.
4. 해당 목록을 카드 리스트 형태로 렌더링한다.

### 11.5 레이아웃 방향

위시리스트는 전체 숙소 페이지보다 데이터 양이 적을 가능성이 높다. 따라서 화면을 과도하게 빽빽하게 채우기보다는, **여백을 충분히 확보한 카드형 배치**가 더 적절하다.

특히 데스크탑에서는 정보량보다 안정적인 화면 균형과 시각적 여유를 우선한다.

## 12. 숙소 상세 페이지 구현 방향

### 12.1 페이지 역할

숙소 상세 페이지는 사용자가 특정 숙소에 대해 예약 여부를 판단하는 핵심 화면이다. 리스트 페이지에서 제공하지 못한 상세 정보를 충분히 제공해야 한다.

### 12.2 표시 정보

| 항목     | 설명              |
| ------ | --------------- |
| 숙소 이미지 | 상세 이미지 갤러리      |
| 위치     | 숙소 지역/주소 정보     |
| 설명     | 숙소 소개           |
| 편의시설   | 제공 서비스 및 설비     |
| 호스트 정보 | 호스트 이름 등        |
| 가격     | 예약 판단 핵심 정보     |
| 리뷰 정보  | 신뢰도 및 만족도 판단 요소 |

### 12.3 예약 인터랙션 개정

기존에는 숙소 상세 페이지에서 일정과 인원을 먼저 선택한 뒤 예약 draft를 생성하는 구조를 고려하였다. 그러나 개정안에서는 이 흐름을 제거한다. 기존 문서의 해당 부분은 draft 생성 중심으로 정리되어 있었다. 

개정 후 숙소 상세 페이지의 예약 관련 역할은 다음과 같다.

* 사용자는 숙소 정보를 확인한다.
* 사용자는 **예약하기** 버튼을 클릭한다.
* 클릭 시 예약 요청 페이지로 이동한다.
* 이 단계에서는 체크인/체크아웃, 게스트 수를 입력하지 않는다.
* 별도의 draft 생성도 수행하지 않는다.

### 12.4 상세 페이지의 예약 관련 책임

| 기능           | 처리 여부 |
| ------------ | ----- |
| 숙소 기본 정보 조회  | 포함    |
| 체크인/체크아웃 선택  | 제외    |
| 게스트 인원 설정    | 제외    |
| draft 생성     | 제외    |
| 예약 요청 페이지 이동 | 포함    |

## 13. 예약 요청 페이지 구현 방향

### 13.1 페이지 역할

예약 요청 페이지는 사용자가 실제 예약에 필요한 정보를 입력하고, 최종 확정하는 화면이다. 즉, 기존에 숙소 상세 페이지가 담당하던 일정 선택 기능을 이 페이지로 이동시킨다.

### 13.2 제공 기능

| 기능                      | 메서드      | 설명                     |
| ----------------------- | -------- | ---------------------- |
| 예약 요청 페이지 진입 시 숙소 정보 조회 | GET      | 선택한 숙소의 예약 대상 정보 조회    |
| 예약 정보 입력                | 클라이언트 입력 | 체크인/체크아웃 날짜, 게스트 인원 설정 |
| 예약 확정                   | POST     | 결제가 완료되었다고 가정하고 예약 생성  |

### 13.3 입력 정보

예약 요청 페이지에서 사용자가 직접 입력하거나 선택해야 하는 정보는 다음과 같다.

| 필드           | 설명           |
| ------------ | ------------ |
| listingId    | 예약 대상 숙소 식별자 |
| checkIn      | 체크인 날짜       |
| checkOut     | 체크아웃 날짜      |
| guests       | 게스트 인원 수     |
| agreeToTerms | 예약 확정 동의 여부  |

### 13.4 수정 기능 제거

기존에는 draft 예약을 조회하고 수정하는 구조를 생각했으나, 개정안에서는 **수정 기능을 제거**한다. 즉, 예약 요청 페이지는 이미 만들어진 예약을 수정하는 곳이 아니라, 예약 정보를 **처음 입력하고 바로 확정하는 화면**이다. 기존 문서의 예약 요청 섹션에는 draft 조회 및 수정 기능이 포함되어 있었으나, 해당 내용은 더 이상 적용되지 않는다. 

### 13.5 날짜 선택 정책

* 지난 날짜는 선택 불가
* 체크아웃은 체크인 이후 날짜만 선택 가능
* 일정 선택 UI는 사용자의 실수를 줄이도록 제약을 명확히 둔다

### 13.6 예약 확정 정책

프로토타입에서는 실제 결제 연동 대신, **결제가 완료되었다고 가정하고 예약을 생성**하는 방식으로 처리한다.

예약 확정이 완료되면 다음 단계로 이동한다.

```md
예약 요청 페이지
→ 예약 확정
→ 예약 상세 페이지 이동
```

### 13.7 예약 확정 후 이동

예약 요청 페이지에서 확정이 완료되면 사용자는 **예약 상세 페이지**로 이동한다.
이 흐름은 사용자가 방금 생성한 예약의 결과를 즉시 확인하도록 하기 위한 것이다.

## 14. 내 예약 확인 페이지 구현 방향

### 14.1 페이지 역할

내 예약 확인 페이지는 사용자가 자신이 확정한 예약을 목록으로 확인하는 화면이다. 로그인 상태에서만 접근할 수 있다.

### 14.2 예약 카드 정보

| 항목            | 설명                 |
| ------------- | ------------------ |
| 숙소 대표 썸네일 이미지 | 예약 숙소 식별           |
| 숙소 이름         | 기본 정보              |
| 예약한 숙박 기간     | 핵심 확인 정보           |
| 찜 상태 하트 아이콘   | 위시리스트와 동일한 상호작용 제공 |

### 14.3 페이지 성격

내 예약 목록은 위시리스트와 유사하게 카드 리스트 기반으로 구성할 수 있다. 이미 랜딩 페이지와 위시리스트 페이지에서 카드 구조를 사용하고 있으므로, 컴포넌트 재사용성 측면에서도 동일한 패턴을 유지하는 것이 유리하다.

## 15. 예약 상세 페이지 구현 방향

### 15.1 페이지 역할

예약 상세 페이지는 사용자가 특정 예약 건의 세부 정보를 확인하고, 필요 시 취소까지 진행할 수 있는 화면이다.

또한 개정된 사용자 흐름에서는 예약 요청 페이지에서 예약을 확정한 직후 처음으로 도달하는 화면이기도 하다.

### 15.2 표시 정보

| 항목                 | 설명               |
| ------------------ | ---------------- |
| 체크인까지 남은 기간        | 체크인 예정일까지 남은 일 수 |
| 체크인 24시간 전부터 남은 시간 | 세밀한 시간 단위 안내     |
| 작은 썸네일 이미지         | 예약 숙소 확인         |
| 숙소 이름              | 기본 식별 정보         |
| 호스트 이름             | 운영 주체 정보         |
| 체크인/체크아웃 날짜 및 시간   | 예약 일정 확인         |
| 게스트 인원 수           | 예약 조건 확인         |
| 총 결제 금액            | 결제 요약 정보         |

### 15.3 제공 기능

* 예약 상세 정보 조회
* 예약 취소

예약 취소는 사용자에게 중요한 제어 기능이므로, 프로토타입에서도 명확한 버튼과 상태 변화를 제공하는 것이 바람직하다.
