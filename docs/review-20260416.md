## 0. 먼저, 좋아진 점부터 볼게요

이번 반영분에서 가장 좋았던 점은, 새로 만든 관리자 숙소 수정 페이지가 기존 `accommodationForm`을 재사용했다는 점이에요.

- 숙소 상세 보기와 수정 화면이 완전히 다른 마크업을 따로 만들지 않고, 공통 폼 컴포넌트를 활용하려고 한 방향은 좋아요.
- 관리자 목록에서 수정 버튼을 추가하고, 상세 페이지에서도 수정 화면으로 자연스럽게 이동할 수 있게 만든 흐름도 사용자 입장에서 분명히 좋아졌어요.
- 버튼 클릭 시 `closest()`를 사용해서 아이콘 클릭까지 안정적으로 처리한 점도 이전보다 안전해졌어요.

다만 이번 변경은 "화면을 붙이는 것"까지는 잘 되었지만, **수정 데이터가 실제 서버에 어떻게 저장되는지**와 **공통 폼과 수정 페이지가 얼마나 느슨하게 연결되어 있는지**에서 중요한 문제가 보여요.

이번 리뷰도 같은 기준으로 볼게요.

> **응집도는 높게**: 한 기능에 필요한 코드는 한곳에 모으기  
> **결합도는 낮게**: 한 파일을 바꿔도 다른 파일이 덜 깨지게 만들기

## 1. 수정 화면에서 "비우기/삭제"가 실제 저장으로 이어지지 않습니다 (중요도: 높음)

**관련 파일:** `admin/modify/index.js:327-379`, `admin/modify/index.js:470-480`, `src/components/accommodationForm.js:1137-1175`, `docs/api-update-fix-20260403.md:98-114`

이번 수정 페이지는 사용자가 값을 비우거나 삭제할 수 있는 UI를 제공해요.

- 썸네일 삭제
- 설명 비우기
- 상세 주소 비우기
- 서비스 수수료 비우기
- 최소 숙박일 비우기

그런데 실제 서버로 보내는 코드를 보면, **비어 있는 값은 요청 본문에서 아예 빠지도록** 되어 있어요.

```js
const description = data.description?.trim() ?? '';
if (description) {
  payload.description = description;
}

const serviceFee = `${data.pricing?.serviceFee ?? ''}`.trim();
if (serviceFee) {
  payload.serviceFee = Number.parseInt(serviceFee, 10);
}
```

썸네일도 같은 문제가 있어요.

```js
const thumbnailUrl = `${modified.thumbnailUrl ?? ''}`.trim();
if (thumbnailUrl && !thumbnailUrl.startsWith('blob:')) {
  requestData.thumbnailUrl = thumbnailUrl;
}
```

즉, 사용자가 화면에서 삭제를 해도 서버에는 이런 식으로 전달될 가능성이 커요.

```js
{
  title: '성수 브릭하우스',
  region: '서울 성동구',
  maxGuest: 4,
  adultPrice: 120000,
  childPrice: 60000
}
```

여기에는 "썸네일을 지웠다", "설명을 비웠다", "서비스 수수료를 없앴다"는 정보가 없어요.

이미 문서에도 이렇게 적혀 있어요.

> 수정할 필드만 보내면 되고, 보내지 않은 필드는 기존 값이 유지됩니다.

즉, 지금 구조에서는 **화면에서는 삭제된 것처럼 보여도, 새로고침하면 다시 살아나는 값**이 생길 수 있어요.

이건 왜 중요할까요?

- 사용자는 "삭제 버튼"을 눌렀는데 실제로는 삭제가 안 돼요.
- 화면 상태와 서버 상태가 달라져요.
- 초보 개발자 입장에서는 "분명 버튼 눌렀는데 왜 안 지워지지?" 같은 디버깅이 아주 어려워져요.

핵심 문제는 이것이에요.

> **"값을 안 보낸 것"과 "값을 지우고 싶다"는 전혀 다른 의미인데, 지금 코드는 둘을 구분하지 못하고 있어요.**

## 2. 이미지와 예약 불가 날짜는 전부 지우는 순간 오히려 원래 값으로 되돌아갈 수 있습니다 (중요도: 높음)

**관련 파일:** `admin/modify/index.js:389-421`, `admin/modify/index.js:483-515`, `src/components/accommodationForm.js:1276-1295`, `src/components/accommodationForm.js:1525-1535`

이번 수정 페이지는 이미지 삭제와 예약 불가 날짜 삭제도 지원해요. 그런데 제출 직전에 데이터를 합치는 코드가 이렇게 되어 있어요.

```js
blockedDates:
  modifiedBlockedDates.length > 0 || originBlockedDates.length === 0
    ? modifiedBlockedDates
    : originBlockedDates,

images:
  modifiedImages.length > 0 || originImages.length === 0
    ? modifiedImages
    : originImages,
```

이 로직은 얼핏 보면 "수정한 값이 있으면 그걸 쓰자"처럼 보여요. 하지만 실제로는 큰 문제가 있어요.

예를 들어 원래 이미지가 3장이 있었고, 사용자가 3장을 모두 삭제했다고 가정해 볼게요.

1. 화면의 `modifiedImages`는 `[]`가 돼요.
2. 그런데 `originImages.length`는 `3`이에요.
3. 위 조건문은 `modifiedImages`가 비어 있으니 `originImages`를 다시 선택해요.

즉, 사용자는 분명 다 지웠는데, 제출 직전 코드가 **"원래 값으로 복구"** 해버려요.

예약 불가 날짜도 똑같아요.

- 사용자가 `HOST_BLOCK` 날짜를 모두 삭제
- `modifiedBlockedDates = []`
- 제출 직전 `originBlockedDates`로 되돌림

그러면 사용자는 "삭제가 안 된다"고 느끼게 돼요.

이건 응집도/결합도 관점에서도 좋지 않아요.

- 삭제 로직은 `accommodationForm` 안에 있어요.
- 그런데 제출 직전 `admin/modify/index.js`가 그 결정을 다시 뒤집고 있어요.

즉, **한 기능의 규칙이 두 군데에 나뉘어 있어요.**

이런 구조는 초보 개발자에게 특히 위험해요. 왜냐하면 나중에 버그가 나면:

- 폼 쪽을 봐야 하는지
- 제출 쪽을 봐야 하는지
- 둘 다 봐야 하는지

판단하기 어려워지기 때문이에요.

## 3. 예약 때문에 막힌 날짜(`RESERVATION`)도 관리자 화면에서 삭제할 수 있습니다 (중요도: 높음)

**관련 파일:** `src/components/accommodationForm.js:1566-1595`, `src/components/accommodationForm.js:1525-1535`, `admin/modify/index.js:254-261`, `admin/modify/index.js:356-375`

예약 불가 날짜는 두 종류가 있어요.

- `HOST_BLOCK`: 관리자가 직접 막은 날짜
- `RESERVATION`: 실제 예약 때문에 막힌 날짜

코드도 이 두 종류를 분명히 알고 있어요.

```js
blockedDate.reason === 'RESERVATION' ? '예약상태' : '관리자지정'
```

그런데 삭제 버튼은 이유를 구분하지 않고 모두 붙여요.

```js
if (mode !== constants.FORM_MODE.VIEW) {
  const button = document.createElement('button');
  button.className = 'blockedDateDeleteBtn ...';
  button.dataset.id = blockedDateId;
  button.textContent = '삭제';
  div.appendChild(button);
}
```

그리고 클릭하면 그냥 지워요.

```js
accommodationForm.removeBlockedDate(blockedDateId);
```

이건 왜 위험할까요?

예약으로 막힌 날짜는 단순 메모가 아니에요. 실제 예약 데이터와 연결된 보호 장치예요. 그런데 지금 UI는 이것도 "관리자 마음대로 지울 수 있는 일반 차단일"처럼 보여주고 있어요.

만약 서버가 프론트에서 보낸 `blockedDates` 배열을 그대로 반영한다면:

1. 예약으로 막혀 있던 날짜를 화면에서 삭제
2. 수정 저장
3. 숙소가 비어 있는 것처럼 다시 열림
4. 같은 날짜에 중복 예약 위험 발생

즉, 이 문제는 단순 UI 문제가 아니라 **예약 무결성**과 연결돼요.

초보 개발자분들은 이런 구분을 꼭 기억하면 좋아요.

- 화면에 비슷하게 보인다고 같은 데이터가 아니에요.
- `HOST_BLOCK`와 `RESERVATION`은 이름이 비슷해도 책임이 달라요.
- 책임이 다른 데이터는 같은 버튼으로 다루면 안 돼요.

## 4. `admin/modify/index.js`가 폼 내부 구조를 너무 많이 알고 있어서 결합도가 높습니다 (중요도: 중간)

**관련 파일:** `admin/modify/index.js:101-268`, `src/components/accommodationForm.js:1601-1724`, `admin/accommodation/index.js:36-67`, `admin/modify/index.js:28-59`

이번 수정 페이지의 가장 큰 구조 문제는, `admin/modify/index.js`가 폼의 세부 구현을 너무 많이 알고 있다는 점이에요.

예를 들면 이 파일은 다음을 전부 직접 알고 있어요.

- `#thumbnailDeleteBtn`
- `#thumbnailInsertBtn`
- `#imageInsertBtn`
- `#imageTitleApplyBtn`
- `#titleApplyBtn`
- `#addressApplyBtn`
- `#maxGuestApplyBtn`
- `#descriptionApplyBtn`
- `#adultPriceApplyBtn`
- `#childPriceApplyBtn`
- `#serviceFeeApplyBtn`
- `#minNightsApplyBtn`
- `#blockedDateAddBtn`

그리고 각각에 대해 `if`를 길게 이어서 처리하고 있어요.

이 구조의 문제는 명확해요.

1. 폼의 버튼 id 하나만 바뀌어도 수정 페이지가 깨져요.
2. `accommodationForm`의 입력 구조가 바뀌면 `admin/modify/index.js`도 같이 수정해야 해요.
3. 결국 "공통 폼"을 재사용했지만, 실제로는 강하게 묶여 있어서 재사용의 장점이 줄어들어요.

이건 낮은 응집도의 전형적인 신호예요.

- 폼 렌더링 책임은 `accommodationForm`
- 클릭 규칙은 `admin/modify/index.js`
- 제출 규칙도 `admin/modify/index.js`
- 뒤로가기 버튼 유틸도 상세 페이지와 수정 페이지에 중복

즉, 하나의 기능이 여러 파일에 흩어져 있어요.

초보 단계에서는 이런 기준으로 보면 좋아요.

> "숙소 수정 기능"을 고치려면 몇 파일을 같이 열어야 하는가?

지금은 너무 많아요. 이런 상태는 나중에 수정할수록 실수할 가능성을 키워요.

## 5. 한 줄 정리

이번 변경은 화면 연결과 재사용 방향은 좋았어요. 하지만 **수정 페이지가 실제 저장 규칙을 아직 완전히 다루지 못하고 있고**, 특히 삭제/비우기/예약 날짜 보호에서 운영 버그가 날 가능성이 큽니다.

가장 먼저 고쳐야 할 순서는 이렇게 추천해요.

1. 비우기/삭제가 서버에 정확히 전달되도록 제출 규칙 정리
2. 이미지/예약 불가 날짜 전체 삭제가 가능한 구조로 수정
3. `RESERVATION` 날짜는 삭제 불가로 보호
4. 수정 페이지 이벤트와 제출 책임을 공통 폼 쪽으로 더 모으기
