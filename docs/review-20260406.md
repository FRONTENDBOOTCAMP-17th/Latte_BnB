## 2. 잘한 점

이번에 추가된 코드의 양과 기능이 정말 많아요. 3일 만에 이렇게 많은 기능을 구현한 건 대단해요!

### 컴포넌트 분리를 잘 했어요

- `hamburger.js`, `navigation.js`, `pagination.js`, `PreviewImage.js`, `numberInput.js` 등 **기능별로 파일을 분리**한 건 아주 좋은 습관이에요. 하나의 파일에 모든 코드를 넣지 않고, 역할에 맞게 나눈 건 **높은 응집도**의 좋은 예시에요!

```
src/components/
├── hamburger.js      ← 햄버거 메뉴만 담당
├── navigation.js     ← 하단 네비게이션만 담당
├── pagination.js     ← 페이지네이션만 담당
├── PreviewImage.js   ← 이미지 프리뷰만 담당
└── numberInput.js    ← 숫자 입력만 담당
```

### RoomCard 클래스의 캡슐화가 훌륭해요

- `#`(private 필드)를 사용해서 외부에서 직접 접근할 수 없게 만든 건 **캡슐화**의 교과서적 사용이에요. `setWish()` 같은 메서드를 통해서만 상태를 바꿀 수 있게 한 건 정말 잘한 설계에요.

```js
// RoomCard.js — private 필드로 데이터 보호
#id;
#title;
#isWish = false;
#wishNode = null;

setWish(isWish) {  // 메서드를 통해서만 상태 변경 가능!
  this.#isWish = isWish;
  // ...
}
```

### 관리자 숙소 등록 페이지의 완성도가 높아요

- `admin/add/`는 이 프로젝트에서 가장 완성도가 높은 페이지에요:
  - `<form>` + `<fieldset>` + `<legend>` 시맨틱 HTML 완벽 사용
  - 이미지 업로드 + 프리뷰 + 삭제 + 모달 편집까지 구현
  - API 호출 로직을 `addAccommodation.js`로 분리한 건 **낮은 결합도**의 좋은 예시에요!
  - 유효성 검사 후 toast로 피드백을 주는 UX도 깔끔해요

```js
// addAccommodation.js — API 호출만 담당 (높은 응집도!)
export async function uploadImage(file) { ... }
export async function addAccommodation(requestData) { ... }
export function buildRequestFormData(form) { ... }
```

### 위시리스트(찜) 기능이 잘 설계되어 있어요

- 페이지 로드 시 `checkWished()`로 일괄 확인하고, 개별 카드 클릭 시 `PATCH` API로 토글하는 구조가 실무적이에요. toast로 성공/실패 피드백을 주는 것도 좋아요!

### 검색 + 정렬 + 페이지네이션이 동작해요

- `landing.js`에서 검색, 정렬, 페이지 이동이 모두 API와 연동되어 동작하는 건 실제 서비스 수준의 구현이에요.

## 3. 개선하면 좋을 점

### 3-1. reservations-check/index.js에 오타 3개가 있어요 — 코드가 동작하지 않아요 (중요도: 매우 높음)

**현재 상태:**

```js
// reservations-check/index.js 25번째 줄
throw new Error('HTTP 오류: ' + res.stauts);
//                                  ^^^^^^ status 오타!

// 68번째 줄
result.sytle.color = 'red';
//     ^^^^^ style 오타!

// 63번째 줄
localStorage.setItem(rsv.status);
//           ^^^^^^^ key가 없어요! setItem(key, value) 형태여야 해요
```

**왜 개선하면 좋을까?**

- `res.stauts`는 `undefined`가 되어 에러 메시지에 "HTTP 오류: undefined"로 표시돼요
- `result.sytle`는 `undefined`라서 `.color`에 접근하면 **즉시 에러가 터져요!**
- `localStorage.setItem(rsv.status)`는 key 없이 value만 넣으면 key가 `undefined`로 저장돼요

**추천:**

```js
// 25번째 줄 수정
throw new Error('HTTP 오류: ' + res.status);

// 68번째 줄 수정
result.style.color = 'red';

// 63번째 줄 수정 — key를 지정하세요
localStorage.setItem('reservationStatus', rsv[i].status);
```

```js
// reservations-check/index.js 29~31번째 줄
```

### 3-4. `res.json()` 파싱을 `res.ok` 확인보다 먼저 하고 있어요 (중요도: 높음)

**현재 상태:**

이 패턴이 `landing.js`, `hamburger.js` 등 여러 곳에서 반복돼요:

```js
// landing.js 21~24번째 줄
const { message, data, meta } = await res.json();  // ① 먼저 JSON 파싱
if (!res.ok) {                                       // ② 그 다음 에러 체크
  throw new Error(message);
}
```

```js
// hamburger.js 98~101번째 줄
const { message, data } = await res.json();  // ① 먼저 JSON 파싱
if (!res.ok) {                                // ② 그 다음 에러 체크
  throw new Error(message);
}
```

**왜 개선하면 좋을까?**

HTTP 에러(500, 404 등)가 나면 응답 본문이 JSON이 아닐 수 있어요. 그러면 `res.json()`에서 **파싱 에러가 먼저 터지고**, `res.ok` 체크까지 도달하지 못해요. 에러 메시지도 "HTTP 오류"가 아니라 "Unexpected token < in JSON"처럼 이상한 메시지가 사용자에게 보여요.

**추천:** `res.ok`를 **먼저** 확인하세요:

```js
if (!res.ok) {
  throw new Error(`HTTP 오류: ${res.status}`);
}
const { message, data, meta } = await res.json();
```

이 패턴을 `landing.js`(3곳), `hamburger.js`(1곳) 모두 수정해주세요.

### 3-5. landing.js의 fetch 함수 3개가 거의 똑같아요 — DRY 원칙 (중요도: 높음)

**현재 상태:**

```js
// landing.js — 3개 함수가 URL 파라미터만 약간 다르고 나머지는 동일해요!
async function fetchRooms() {
  const res = await fetch(`${API}/accommodations?pageLimit=${pageLimit}&sort=${sortBox.value}`);
  const { message, data, meta } = await res.json();
  if (!res.ok) throw new Error(message);
  return { data, meta };
}

async function fetchPage() {
  const res = await fetch(`${API}/accommodations?page=${pagination.paginationData.page}&pageLimit=${pageLimit}&sort=${sortBox.value}`);
  const { message, data, meta } = await res.json();  // 완전 같은 코드!
  if (!res.ok) throw new Error(message);              // 완전 같은 코드!
  return { data, meta };                              // 완전 같은 코드!
}

async function fetchSearch(query) {
  const res = await fetch(`${API}/accommodations?pageLimit=${pageLimit}&query=${query}&sort=${sortBox.value}`);
  const { message, data, meta } = await res.json();  // 또 같은 코드!
  if (!res.ok) throw new Error(message);
  return { data, meta };
}
```

**왜 개선하면 좋을까?**

**DRY(Don't Repeat Yourself)** 원칙은 "같은 코드를 반복하지 말자"예요. 지금 3개 함수의 구조가 거의 동일한데, 만약 에러 처리 방식을 바꾸고 싶으면 3곳을 모두 수정해야 해요. 빠뜨리면 일부만 수정되어 버그가 생겨요.

이건 **응집도는 높이고, 결합도는 낮추는** 대표적인 리팩토링 포인트에요. 공통 로직을 하나의 함수로 묶으면, 각 호출 지점은 "무엇을 가져올지"만 신경 쓰면 돼요.

**추천:** 하나의 범용 함수로 합치세요:

```js
async function fetchAccommodations({ page, query } = {}) {
  const params = new URLSearchParams({
    pageLimit,
    sort: sortBox.value,
  });
  if (page) params.set('page', page);
  if (query) params.set('query', query);

  const res = await fetch(`${constants.API_BASE_URL}/accommodations?${params}`);
  if (!res.ok) throw new Error(`HTTP 오류: ${res.status}`);

  const { data, meta } = await res.json();
  return { data, meta };
}

// 사용:
const result = await fetchAccommodations();                          // 전체 조회
const result = await fetchAccommodations({ page: 2 });               // 페이지 이동
const result = await fetchAccommodations({ query: '서울' });          // 검색
```

### 3-6. landing.js에 try/catch가 없어서 에러 시 화면이 멈춰요 (중요도: 높음)

**현재 상태:**

```js
// landing.js 110~114번째 줄 — 최상위 레벨에서 await를 바로 사용
const result = await fetchRooms();       // ← 에러 나면?
```

```js
// landing.js 103~108번째 줄 — changePage도 에러 처리 없음
async function changePage() {
  const result = await fetchPage();   // ← API 실패 시 크래시!
  // ...
```

**왜 개선하면 좋을까?**

네트워크 오류, 서버 다운, 토큰 만료 등의 상황에서 `fetch`가 실패하면, try/catch가 없으면 에러가 **잡히지 않고** 나머지 코드가 실행되지 않아요. 사용자는 빈 화면만 보게 돼요.

```js
}
```

### 3-7. reservations-check에서 API URL을 하드코딩하고 있어요 (중요도: 높음)

**현재 상태:**

```js
// reservations-check/index.js 13~14번째 줄
const res = await fetch(
```

### 3-14. header.js에서 Promise 에러 처리가 없어요 (중요도: 중간)

**현재 상태:**

```js
// src/components/header.js 37~45번째 줄
const profilePromise = hamburger.checkAuth();

profilePromise.then((result) => {
  header.getElementsByClassName('hamburger')[0]
    .appendChild(hamburger.buildMenu(result));
  hamburger.attachEvent();
  navigation.buildMenu(result);
});
// ← .catch()가 없어요!
```

**왜 개선하면 좋을까?**

`hamburger.checkAuth()`가 실패하면 (네트워크 오류 등), Promise가 거부되는데 `.catch()`가 없으면 **Unhandled Promise Rejection** 경고가 콘솔에 나와요. 그리고 햄버거 메뉴와 네비게이션이 **아예 표시되지 않아요**.

**추천:**

```js
profilePromise
  .then((result) => {
    header.getElementsByClassName('hamburger')[0]
      .appendChild(hamburger.buildMenu(result));
    hamburger.attachEvent();
    navigation.buildMenu(result);
  })
  .catch((error) => {
    console.error('인증 확인 실패:', error);
    // 비로그인 상태의 메뉴라도 보여주기
    const defaultResult = { isAuth: false, data: null };
    header.getElementsByClassName('hamburger')[0]
      .appendChild(hamburger.buildMenu(defaultResult));
    hamburger.attachEvent();
    navigation.buildMenu(defaultResult);
  });
```

## 7. 마무리

이번 리뷰 기간 동안 프로젝트가 정말 크게 성장했어요!

**특히 인상적인 점:**
- **관리자 숙소 등록 페이지**는 이 프로젝트에서 가장 완성도가 높아요. `<form>`, `<fieldset>`, `<legend>` 시맨틱 태그, 이미지 업로드/프리뷰, API 호출 분리까지 모범적이에요
- **RoomCard 클래스**의 `#private` 필드 활용과 캡슐화는 초보 수준을 넘어서는 설계에요
- **위시리스트 기능**의 일괄 체크 + 개별 토글 구현은 실제 서비스에서 쓰이는 패턴이에요
- **컴포넌트 분리**(hamburger, navigation, pagination, PreviewImage, numberInput)를 통해 **응집도를 높이려는 노력**이 확실히 보여요

**가장 시급한 건** `reservations-check/index.js`의 **오타 3개와 데이터 구조 버그**에요. 이 4개만 수정하면 예약 확인 페이지가 정상 동작할 거에요. 그 다음은 `landing.js`의 **fetch 중복 제거**가 좋은 리팩토링 연습이 될 거예요.

한 가지 아쉬운 점은, **이전 리뷰에서 지적한 오타**(RoomCard의 `col-satrt-1`, `tumbnailUrl`)가 2차 리뷰부터 3번째 리뷰인데도 아직 수정되지 않았어요. 오타는 찾아서 바꾸기(Ctrl+H)로 **1분이면 수정 가능**하니 꼭 해주세요!

전체적으로 **응집도는 높이려는 좋은 시도**들이 보이고, **결합도를 낮추는 방향**으로 constants.js, addAccommodation.js 같은 모듈 분리를 잘 하고 있어요. 이 방향을 유지하면서 나머지 페이지에도 같은 패턴을 적용하면 프로젝트 품질이 한 단계 더 올라갈 거예요. 화이팅!
