## 2. 잘한 점

### 2-1. landing.js 리팩터링이 깔끔해요

이전 리뷰에서 `fetchPage`, `fetchSearch`, `fetchRooms`로 나뉘어 있던 세 개의 함수를 하나의 `fetchAccommodations()`로 통일했어요. 이것이 바로 **응집도를 높인** 좋은 예시입니다.

```js
// 하나의 함수로 페이지, 검색, 정렬을 모두 처리
async function fetchAccommodations({ page, query } = {}) {
  const params = new URLSearchParams({ pageLimit, sort: sortBox.value });
  if (page) params.set('page', page);
  if (query) params.set('query', query);
  // ...
}
```

`URLSearchParams`를 활용해서 조건부로 파라미터를 추가하는 방식이 매우 깔끔합니다.

### 2-2. header.js의 검색바 분리가 훌륭해요

`buildHeader()`와 `buildSearchBar()`를 분리한 것은 **단일 책임 원칙(SRP)**을 잘 지킨 것이에요. 검색바가 필요한 페이지와 아닌 페이지를 `data-search` 속성으로 구분하는 설계가 좋습니다.

```html
<!-- 검색바가 필요한 페이지만 -->
<body data-header="true" data-search="true">
```

### 2-3. main.js의 data 속성 기반 컴포넌트 조립이 멋져요

`data-header`, `data-footer`, `data-nav`, `data-search` 속성으로 페이지마다 필요한 컴포넌트만 조립하는 패턴이 정말 좋아요. 이건 **결합도를 낮추는** 훌륭한 설계입니다.

```js
if (document.body.dataset.header === 'true') {
  document.body.prepend(header.buildHeader());
}
if (document.body.dataset.search === 'true') {
  document.querySelector('header').insertBefore(header.buildSearchBar(), ...);
}
```

### 2-4. RoomCard 클래스의 private 필드 활용이 좋아요

`#id`, `#title` 등 private 필드를 사용해서 외부에서 직접 접근할 수 없게 만든 것은 **캡슐화**를 잘 지킨 것이에요. 데이터를 변경할 때는 반드시 `set()` 메서드를 통해야 하고, DOM도 `#syncDOM()`으로 자동 업데이트되는 구조가 훌륭합니다.

### 2-5. 예약 신청 페이지의 캘린더가 잘 만들어졌어요

`calendar.js`에서 체크인/체크아웃 날짜 선택, 차단된 날짜 처리, 날짜 범위 하이라이트 등 복잡한 로직을 직접 바닐라 JS로 구현한 것이 대단해요. 특히 `isBlocked()` 함수로 예약 불가 날짜를 검사하는 로직이 잘 작성되었습니다.

### 2-6. try/catch 예외처리를 전면 적용한 점이 좋아요

이전 리뷰에서 지적된 대로 `async/await` 사용 부분에 모두 `try/catch`를 추가했어요. 에러가 발생해도 페이지가 멈추지 않고 사용자에게 알려주는 안정적인 코드가 되었습니다.

### 2-7. admin/add에 ObjectURL 메모리 해제를 추가한 점이 좋아요

`URL.revokeObjectURL()`을 호출해서 더 이상 사용하지 않는 미리보기 이미지의 메모리를 해제하는 코드를 추가한 건 메모리 누수를 방지하는 좋은 습관이에요.

### 2-8. profile 페이지의 아코디언 구현이 깔끔해요

Tailwind의 `group-[.active]` 클래스를 활용한 아코디언 토글이 CSS만으로 애니메이션까지 처리하는 깔끔한 구현이에요.

## 3. 개선하면 좋을 점

### 3-1. reservations-check/index.js에 심각한 버그가 있어요 (중요도: 높음)

이 파일에는 **3가지 버그**가 있어서 페이지가 제대로 동작하지 않을 수 있어요.

**현재 상태:**
```js
// 버그 1: 오타 — res.stauts → res.status
throw new Error('HTTP 오류: ' + res.stauts);   // 25행

// 버그 2: 배열의 개별 항목이 아닌 배열 자체에서 속성 접근
const rsv = rsvData.data.reservations;    // rsv는 배열
const amdData = rsv.accommodation;        // undefined! 배열에는 .accommodation가 없음
const schData = rsv.schedule;             // undefined! 마찬가지

// 버그 3: localStorage.setItem은 인자가 2개 필요
localStorage.setItem(rsv.id);             // key만 있고 value가 없음
localStorage.setItem(rsv.status);         // 같은 문제

// 버그 4: 오타 — result.sytle → result.style
result.sytle.color = 'red';               // 69행
```

**왜 개선하면 좋을까?**
이 버그들 때문에 예약 목록 페이지가 아예 동작하지 않아요. `rsv`가 배열인데 `.accommodation`을 직접 접근하면 `undefined`가 되고, 반복문 안에서 `amdData[i]`를 하면 `undefined[i]`가 되어 에러가 터집니다.

**추천:**
```js
// 수정된 코드
if (!res.ok) {
  throw new Error('HTTP 오류: ' + res.status);  // stauts → status
}

const rsvData = await res.json();
const rsv = rsvData.data.reservations;

if (rsv.length === 0) {
  result.textContent = '예약 내역이 없습니다.';
  return;
}

for (let i = 0; i < rsv.length; i++) {
  const amd = rsv[i].accommodation;   // 개별 예약에서 숙소 정보 꺼내기
  const sch = rsv[i].schedule;        // 개별 예약에서 일정 정보 꺼내기

  // ... 카드 생성 코드 ...

  localStorage.setItem('reservation_id', rsv[i].id);     // key, value 2개
  localStorage.setItem('reservation_status', rsv[i].status);
}

// catch 블록에서
result.style.color = 'red';   // sytle → style
```

### 3-2. reservations-detail/index.html에서 API URL 하드코딩 + 추가 버그 (중요도: 높음)

**현재 상태:**
```js
// 하드코딩된 API URL
const res = await fetch(
  `https://api.fullstackfamily.com/api/lattebnb/v1/reservations/${reservationId}`,

// location.href를 함수처럼 호출 — TypeError 발생!
location.href(`../reservations-check`);   // 261행
```

**왜 개선하면 좋을까?**
1. API URL이 바뀌면 이 파일을 찾아서 일일이 수정해야 해요. `constants.js`에 이미 `API_BASE_URL`이 있으니 그걸 쓰면 한 곳만 바꾸면 돼요. 이것이 **결합도를 낮추는** 핵심이에요.
2. `location.href`는 **속성(문자열)**이지 함수가 아닙니다. `location.href(...)`로 호출하면 `TypeError: location.href is not a function` 에러가 발생해요.
3. 이 페이지는 `<script type="module">`이 아닌 일반 `<script>` 태그를 사용하고 있어 `import`를 사용할 수 없어요. 모듈 방식으로 전환해야 합니다.

**추천:**
```html
<!-- index.html에서 script를 모듈로 변경 -->
<script type="module" src="./index.js"></script>
```

```js
// index.js (별도 파일로 분리)
import constants from '../src/constants.js';

const API_BASE = constants.API_BASE_URL;

// API 호출 시
const res = await fetch(`${API_BASE}/reservations/${reservationId}`, { ... });

// 페이지 이동 시 — 속성에 값을 할당
location.href = '/reservations-check/';   // = 으로 할당!
```

### 3-3. 인증 체크 로직이 여러 곳에 중복돼요 (중요도: 높음)

**현재 상태:**
같은 "토큰으로 프로필 조회해서 인증 확인" 로직이 최소 5곳에 반복됩니다.

```js
// main.js (42행)
async function checkAuth() {
  const token = localStorage.getItem('accessToken');
  const data = await getProfile(token);
  ...
}

// wishlist/index.js (7~28행)
const token = localStorage.getItem('accessToken');
const res = await fetch(`${API_BASE}/me/profile`, { ... });

// reservation-request.js (8~29행)
async function checkTokenAvailable() {
  if (!token) { ... }
  const res = await fetch(`${API_BASE}/me/profile`, { ... });
  ...
}

// profile/profile.js (8~13행)
if (!token) {
  alert('로그인을 먼저 해주세요');
  location.href = '/login/';
}

// admin/add/index.js (16~36행)
const res = await fetch(`${API_BASE}/me/profile`, { ... });
```

**왜 개선하면 좋을까?**
동일한 코드가 5곳에 있으면 버그를 고칠 때도 5곳을 모두 수정해야 해요. 하나라도 빠뜨리면 버그가 남아있게 됩니다. 이것이 바로 **결합도가 높은** 상태에요.

**추천:**
`src/utils/auth.js` 같은 공통 모듈을 만들어서 한 곳에서 관리하세요.

```js
// src/utils/auth.js — 새로 만들 파일
import constants from '../constants.js';

/**
 * 토큰 유효성을 확인합니다.
 * 유효하면 사용자 데이터를 반환하고, 유효하지 않으면 null을 반환합니다.
 */
export async function checkAuth() {
  const token = localStorage.getItem('accessToken');
  if (!token) return null;

  const res = await fetch(`${constants.API_BASE_URL}/me/profile`, {
    method: 'GET',
    headers: { Authorization: `Bearer ${token}` },
  });

  if (!res.ok) return null;

  const { data } = await res.json();
  return data;
}

/**
 * 로그인이 필요한 페이지에서 호출합니다.
 * 인증 실패 시 로그인 페이지로 이동합니다.
 */
export async function requireAuth() {
  const user = await checkAuth();
  if (!user) {
    alert('로그인이 필요합니다.');
    location.href = '/login/';
    return null;
  }
  return user;
}
```

```js
// 사용하는 쪽 (예: wishlist/index.js)
import { requireAuth } from '../src/utils/auth.js';

const user = await requireAuth();  // 인증 실패 시 자동으로 로그인 페이지로 이동
if (!user) throw new Error('인증 필요');
// 이후 로직 계속...
```

### 3-4. calendar.js가 전역 변수에 의존해요 (중요도: 중간)

**현재 상태:**
```js
// calendar.js — 모든 변수가 전역
let checkin = null;
let checkout = null;
let blockedDates = [];
let currentMonth = new Date();
let onCalendarChange = null;

// reservation-request.js에서 전역 변수를 직접 조작
onCalendarChange = updateDateBtn;   // 35행
blockedDates = room.bookingPolicy.blockedDates;  // 96행
```

`calendar.js`가 `type="module"`이 아닌 일반 스크립트로 로드되어 있어서, `checkin`, `checkout`, `blockedDates` 같은 변수들이 **전역(window)**에 노출됩니다. `reservation-request.js`는 모듈인데 전역 변수를 직접 읽고 쓰고 있어요.

**왜 개선하면 좋을까?**
전역 변수는 어디서든 변경할 수 있어서 버그 추적이 어렵습니다. 다른 스크립트가 실수로 `checkin`이라는 이름을 쓰면 충돌이 일어나요. 이것이 **높은 결합도**의 전형적인 예시에요.

**추천:**
`calendar.js`도 ES 모듈로 바꾸고, 필요한 것만 export하세요.

```js
// calendar.js — 모듈로 변환
let checkin = null;
let checkout = null;
let blockedDates = [];
// ...

export function getCheckinDate() { ... }
export function getCheckoutDate() { ... }
export function setBlockedDates(dates) { blockedDates = dates; }
export function setOnChange(callback) { onCalendarChange = callback; }
export function buildCalendar() { ... }

// 기본 초기화는 모듈 로드 시 실행하지 않고, 호출하는 쪽에서 명시적으로 init
export function init() { buildCalendar(); }
```

```html
<!-- reservation-request/index.html -->
<!-- 둘 다 모듈로 변경 -->
<script type="module" src="./reservation-request.js"></script>
<!-- calendar.js는 reservation-request.js에서 import -->
```

```js
// reservation-request.js
import { setBlockedDates, setOnChange, getCheckinDate, getCheckoutDate, init as initCalendar }
  from './calendar.js';

setOnChange(updateDateBtn);
// ...
setBlockedDates(room.bookingPolicy.blockedDates);
initCalendar();
```

```js
// wishlist/index.js (135~147행)
document.addEventListener('click', async (e) => {
  if (e.target.classList.contains('wishHeart')) {
    const id = Number.parseInt(e.target.closest('.accommodationCard').dataset.id);
    const target = roomData.get(id);
    target.getElement().remove();   // DOM에서만 제거
    roomData.delete(id);            // 로컬 Map에서만 제거

    }

```

**왜 개선하면 좋을까?**
하트를 클릭하면 화면에서는 사라지지만, **서버에는 여전히 찜 상태**로 남아있어요. 새로고침하면 다시 나타납니다. `RoomCard` 내부의 `wishHeart` 클릭 이벤트가 실행되면 서버 API를 호출하긴 하지만, 위시리스트 페이지에서의 이벤트 위임(`wishHeart` 클래스 체크)과 `RoomCard` 내부 이벤트가 충돌할 수 있어요.

**추천:**
`RoomCard` 내부 이벤트가 서버 동기화를 처리하므로, 위시리스트 페이지에서는 서버 응답 이후에 DOM을 정리하면 됩니다. 또는 `RoomCard`에 콜백 옵션을 추가하는 것도 좋아요.

```js
// 간단한 방법: RoomCard 내부 이벤트가 끝난 후 DOM 정리
// wishHeart 이벤트 대신, RoomCard에서 wish 상태가 변경되었을 때 알려주는 방식
document.addEventListener('click', async (e) => {
  if (e.target.classList.contains('wishHeart')) {
    // RoomCard 내부 이벤트가 서버 API를 호출하니 약간 대기
    setTimeout(() => {
      const id = Number.parseInt(e.target.closest('.accommodationCard').dataset.id);
      const target = roomData.get(id);
      if (target && !target.isWished()) {  // 찜 해제되었으면
        target.getElement().remove();
        roomData.delete(id);
        if (roomData.size === 0) renderEmpty();
      }
    }, 500);
  }
});
```

### 3-6. 검색 이벤트에서 searchBtn 클릭이 발생 안 할 수 있어요 (중요도: 중간)

**현재 상태:**
```js
// landing.js (116~137행)
document.addEventListener('click', async (e) => {
  if (e.target.id === 'searchBtn') {   // searchBtn의 id를 가진 요소를 클릭해야 동작
    // ...
  }
});
```

```html
<!-- header.js에서 생성한 검색 버튼 -->
<button id="searchBtn" type="button">
  <svg class="pointer-events-none">...</svg>  <!-- SVG에는 pointer-events-none -->
</button>
```

SVG에 `pointer-events-none`이 있어서 SVG를 클릭하면 이벤트가 버튼으로 전파되어 동작하긴 합니다. 하지만 만약 버튼 내부에 `pointer-events-none`이 없는 다른 자식 요소가 추가되면 깨질 수 있어요.

**추천:**
`e.target.id` 대신 `e.target.closest('#searchBtn')`을 사용하면 자식 요소를 클릭해도 안전합니다.

```js
document.addEventListener('click', async (e) => {
  const searchBtn = e.target.closest('#searchBtn');
  if (searchBtn) {
    // 검색 실행
  }
});
```

### 3-7. reservations-detail 페이지에서 공통 컴포넌트를 사용하지 않아요 (중요도: 중간)

```html
    // CSS도 커스텀 <style> 태그로 작성
  </script>
</body>
```

**왜 개선하면 좋을까?**
다른 페이지들은 모두 `main.js`를 통해 공통 헤더/푸터/네비게이션을 쓰는데, 이 페이지만 따로 구현하고 있어요. 사용자 경험이 일관되지 않고, 스타일도 다른 페이지와 달라집니다. Tailwind CSS도 사용하지 않고 있어 프로젝트 전체의 일관성이 깨져요.

**추천:**
1. 인라인 `<script>`를 `index.js`로 분리하고 `type="module"`로 로드
2. `main.js`를 활용해서 공통 컴포넌트 적용
3. 커스텀 CSS를 Tailwind 클래스로 교체

```html
<!-- reservations-detail/index.html -->
<body data-header="true" data-footer="true" data-nav="true">
  <script type="module" src="../src/main.js"></script>
  <script type="module" src="./index.js"></script>
  <!-- ... -->
</body>
```

### 3-8. accommodations-detail에서 XSS 취약점 가능성 (중요도: 중간)

**현재 상태:**
```js
// accommodations-detail.js (34~49행)
figure.innerHTML =
  `<img src="${img.url}" alt="${img.title || '숙소 이미지'}" ... />` +
  `<span class="...">${img.title || ''}</span>` +
  `<span class="...">${img.description || ''}</span>`;
```

API에서 받아온 `img.title`, `img.description`을 `innerHTML`로 직접 삽입하고 있어요.

**왜 개선하면 좋을까?**
만약 누군가 숙소 이미지 제목에 `<script>alert('해킹')</script>` 같은 코드를 넣으면, 그게 그대로 실행될 수 있어요. 이것을 **XSS(Cross-Site Scripting)** 공격이라고 합니다.

**추천:**
`innerHTML` 대신 `textContent`를 사용하거나, 구조를 먼저 만들고 텍스트를 따로 넣으세요.

```js
const figure = document.createElement('figure');
figure.className = 'min-w-50 snap-start m-0';

const imgEl = document.createElement('img');
imgEl.src = img.url;
imgEl.alt = img.title || '숙소 이미지';
imgEl.className = 'w-full aspect-4/3 object-cover rounded-lg';

const caption = document.createElement('figcaption');
caption.className = 'flex flex-col mt-1.5';

const titleSpan = document.createElement('span');
titleSpan.className = 'text-sm font-semibold text-shark-800';
titleSpan.textContent = img.title || '';

const descSpan = document.createElement('span');
descSpan.className = 'text-[13px] text-shark-500';
descSpan.textContent = img.description || '';

caption.append(titleSpan, descSpan);
figure.append(imgEl, caption);
gallery.appendChild(figure);
```
