## 수정 순서 추천

이번 이슈들은 순서가 중요해요. 아래 순서대로 고치면 중간에 덜 꼬입니다.

1. **이슈 1** 비우기/삭제 전송 규칙 정리
2. **이슈 2** 이미지/예약 불가 날짜 전체 삭제 가능하게 수정
3. **이슈 3** `RESERVATION` 날짜 보호
4. **이슈 4** 수정 페이지와 폼의 결합도 낮추기

이 순서를 추천하는 이유는, 먼저 **서버에 어떤 데이터를 보낼지**가 정리되어야 그 다음 UI와 구조도 안정적으로 정리할 수 있기 때문이에요.

## 이슈 1. "안 보냄"과 "비움"을 구분해야 합니다

### 왜 고쳐야 하나요

현재 수정 페이지는 사용자가 값을 지워도, 비어 있는 값은 요청 본문에서 빠져버립니다.

그런데 이 프로젝트의 수정 API는 "보내지 않은 값은 기존 값 유지" 방식이에요. 즉:

- 값을 안 보냄 = 그대로 둠
- 빈 값으로 보냄 = 비우고 싶음

이 둘은 완전히 다른 뜻이에요.

### 목표

- 사용자가 값을 수정하지 않았을 때만 필드를 생략하기
- 사용자가 지우기를 선택했을 때는 "지우기 의도"를 명확히 보내기

### 먼저 역할부터 분리하세요

수정 화면에는 사실 3가지 데이터가 있어요.

1. `originData`: 서버에서 처음 받은 원본 데이터
2. `modifiedData`: 사용자가 화면에서 수정한 현재 데이터
3. `requestData`: 서버로 최종 전송할 데이터

이 3개가 섞이면 버그가 쉽게 생겨요.

### 추천 방법

각 필드마다 "사용자가 건드렸는지"를 같이 관리하세요.

예시:

```js
const touched = {
  description: false,
  thumbnailUrl: false,
  serviceFee: false,
  minNights: false,
};
```

그리고 버튼을 눌러 실제 수정이 일어났을 때만 `true`로 바꾸세요.

```js
function updateAccommodationDescription(description) {
  modifiedData.description = description.trim();
  touched.description = true;
}
```

그 다음 제출할 때는 이렇게 생각하면 쉬워요.

- `touched.description === false` 이면 아예 보내지 않음
- `touched.description === true` 이고 값이 비었으면 빈 값으로 보냄

### 주의할 점

문자열과 숫자는 "비우는 방식"이 다를 수 있어요.

- `description`, `address`, `thumbnailUrl` 같은 문자열
- `serviceFee`, `minNights` 같은 숫자

숫자는 백엔드가 무엇을 삭제 의미로 받는지 먼저 확인해야 해요.

- `null`
- `0`
- 빈 문자열

이건 반드시 API 규약을 먼저 맞추고 가야 해요.

### 핵심 정리

> "필드가 없음"은 수정 안 함이고, "빈 값이 들어 있음"은 삭제일 수 있어요.

이 차이를 코드에서 분리해야 합니다.

## 이슈 2. 배열 데이터는 빈 배열도 정상적인 결과로 인정해야 합니다

### 왜 고쳐야 하나요

현재 코드는 이미지나 예약 불가 날짜를 전부 삭제하면, 제출 직전에 원래 값으로 되돌립니다.

```js
images:
  modifiedImages.length > 0 || originImages.length === 0
    ? modifiedImages
    : originImages
```

이 구조는 "빈 배열은 잘못된 값"이라고 가정하고 있어요. 하지만 실제로는:

- 이미지 0장
- 예약 불가 날짜 0개

도 충분히 가능한 상태예요.

즉, 빈 배열은 오류가 아니라 **유효한 사용자 선택**일 수 있어요.

### 목표

- 사용자가 다 지웠으면 서버에도 빈 배열을 보낼 수 있게 하기
- 폼에서 삭제한 결과를 제출 단계가 다시 뒤집지 않게 하기

### 추천 수정 방향

`mergeAccommodationData()`에서 배열을 복구하지 마세요.

```js
bookingPolicy: {
  ...(origin?.bookingPolicy ?? {}),
  ...(modified?.bookingPolicy ?? {}),
},
images: modifiedImages,
```

그리고 요청 본문을 만들 때도 "비어 있으면 생략"이 아니라, **사용자가 수정한 배열이면 그대로 보내기**가 더 안전해요.

예시:

```js
if (touched.images) {
  payload.images = uploadedImages;
}

if (touched.blockedDates) {
  payload.blockedDates = sanitizedBlockedDates;
}
```

여기서 중요한 것은 `length > 0`가 아니에요.  
중요한 것은 **사용자가 이 필드를 수정했는가**예요.

### 초보 개발자용 체크 기준

배열을 다룰 때는 항상 이렇게 질문해 보세요.

> "빈 배열은 버그인가, 아니면 사용자의 정상 선택인가?"

이번 경우는 후자예요.

## 이슈 3. `RESERVATION` 날짜는 화면에서 삭제 못 하게 막아야 합니다

### 왜 고쳐야 하나요

`HOST_BLOCK`는 관리자가 만든 차단일이라 수정 가능할 수 있어요.  
하지만 `RESERVATION`은 실제 예약에서 나온 차단일이에요.

이 둘을 같은 "삭제 버튼"으로 다루면 안 됩니다.

### 목표

- `HOST_BLOCK`만 삭제 가능하게 만들기
- `RESERVATION`은 읽기 전용으로 보여주기

### 가장 쉬운 방법

삭제 버튼을 만들 때 이유를 검사하세요.

```js
if (
  mode !== constants.FORM_MODE.VIEW &&
  blockedDate.reason !== 'RESERVATION'
) {
  // 삭제 버튼 생성
}
```

그리고 예약으로 생성된 날짜는 문구를 더 분명하게 보여주세요.

예시:

- `관리자 지정 차단일`
- `예약으로 자동 차단된 날짜`

이렇게 표시하면 사용자도 헷갈리지 않아요.

### 한 단계 더 좋은 방법

`removeBlockedDate()`에서도 한 번 더 막아 주세요.

```js
function removeBlockedDate(blockedDateId) {
  const blockedDate = blockedDateMap.get(blockedDateId);
  if (!blockedDate) return;
  if (blockedDate.reason === 'RESERVATION') return;

  blockedDateMap.delete(blockedDateId);
}
```

이런 방어 코드는 중요해요.  
UI에서 한 번 막고, 로직에서도 한 번 더 막으면 실수를 줄일 수 있어요.

## 이슈 4. 수정 페이지의 이벤트와 제출 책임을 폼 쪽으로 더 모으세요

### 왜 고쳐야 하나요

지금 `admin/modify/index.js`는 폼의 버튼 id와 입력 id를 너무 많이 알고 있어요. 그래서 폼 구조가 바뀌면 수정 페이지도 같이 깨집니다.

이건 곧:

- 응집도 낮음
- 결합도 높음

상태예요.

### 목표

- "숙소 수정 화면"의 규칙을 더 적은 파일에서 이해할 수 있게 만들기
- 폼 구조가 바뀌어도 바깥 페이지가 덜 흔들리게 만들기

### 추천 구조 1: 액션 맵 사용

긴 `if` 체인을 줄이고, 버튼 id와 처리 함수를 표처럼 관리할 수 있어요.

```js
const clickActions = {
  titleApplyBtn: () => {
    accommodationForm.updateAccommodationTitle(
      content.querySelector('#titleInput')?.value ?? '',
    );
  },
  descriptionApplyBtn: () => {
    accommodationForm.updateAccommodationDescription(
      content.querySelector('#descriptionInput')?.value ?? '',
    );
  },
};
```

이 방식의 장점:

- 버튼이 늘어나도 구조가 덜 복잡해져요.
- 어떤 버튼이 어떤 함수를 부르는지 한눈에 보여요.

### 추천 구조 2: 폼 객체가 제출 데이터를 만들게 하기

지금은 `admin/modify/index.js`가:

- 원본 데이터 읽기
- 수정 데이터 읽기
- 비교
- 병합
- 업로드
- 요청 본문 만들기

를 거의 다 알고 있어요.

더 좋은 구조는, 이런 규칙을 `accommodationForm` 안으로 조금 더 모으는 거예요.

예를 들면:

```js
const submitData = await accommodationForm.buildSubmitPayload();
```

이렇게 되면 바깥 페이지는:

- 페이지 진입
- 저장 버튼 클릭
- 성공 시 이동

정도만 담당하면 돼요.

### 추천 구조 3: 공통 버튼 유틸은 별도 파일로 분리

뒤로가기 버튼 생성 함수가 상세 페이지와 수정 페이지에 거의 똑같이 들어가 있어요.

이럴 때는 CSS 파일을 늘리지 말고, **Tailwind 클래스 문자열을 유지한 채 JS 유틸 함수**로 빼는 것이 좋아요.

예시:

```js
// src/components/historyBackButton.js
export function buildHistoryBackButton(fallbackHref) {
  // 공통 로직
}
```

이렇게 하면 CSS를 추가하지 않아도 되고, Tailwind 유틸 중심 방향도 유지할 수 있어요.

## 추천 작업 순서 예시

1. `touched` 상태를 설계한다.
2. 문자열/숫자/배열별로 "삭제 의미"를 백엔드와 맞춘다.
3. `mergeAccommodationData()`의 배열 복구 로직을 제거한다.
4. `RESERVATION` 날짜는 삭제 버튼을 숨기고, 로직에서도 삭제를 막는다.
5. `admin/modify/index.js`의 긴 이벤트 분기를 액션 맵 또는 폼 메서드로 정리한다.
6. 공통 뒤로가기 버튼 유틸을 분리한다.
