## 이슈 1. `modal.js`는 "호출 횟수"가 아니라 "실제 열림 상태"를 기준으로 동작해야 합니다

### 왜 고쳐야 하나요

지금 코드는 `openModal()`을 부르면 무조건 카운트를 올리고, `closeModal()`을 부르면 무조건 카운트를 내려요. 하지만 이미 닫힌 모달에 `closeModal()`을 호출하면 실제 상태는 안 바뀌는데 카운트만 줄어들어요.

공통 유틸은 이런 식으로 "호출하는 쪽이 실수하지 않아야만 안전한 구조"가 되면 안 돼요. 공통 유틸은 가능한 한 스스로 안전해야 해요.

### 목표

- 이미 열린 요소를 다시 열어도 상태가 꼬이지 않게 만들기
- 이미 닫힌 요소를 다시 닫아도 상태가 꼬이지 않게 만들기
- `body`의 `overflow-hidden`은 실제로 열린 모달이 있을 때만 유지하기

### 수정 방법 1: 가장 간단한 방어 코드 추가

`src/components/modal.js`

```js
let openCount = 0;

export function openModal(element) {
  if (!element) return;
  if (!element.classList.contains('hidden')) return;

  element.classList.remove('hidden');
  element.classList.add('flex');
  openCount++;
  document.body.classList.add('overflow-hidden');
}

export function closeModal(element) {
  if (!element) return;
  if (element.classList.contains('hidden')) return;

  element.classList.remove('flex');
  element.classList.add('hidden');
  openCount = Math.max(0, openCount - 1);

  if (openCount === 0) {
    document.body.classList.remove('overflow-hidden');
  }
}
```

이것만 해도 지금 보이는 버그는 대부분 막을 수 있어요.

### 수정 방법 2: 더 좋은 방향

사실 더 좋은 방식은 `openCount` 같은 전역 숫자에 의존하지 않는 거예요. 예를 들면:

- 오버레이 모달이 열리면 `body` 잠금
- 오버레이 모달이 모두 닫히면 `body` 잠금 해제

즉, 패널(`cancelModal`)과 배경(`modalContainer`)을 같은 기준으로 세지 않는 거예요.

초보 단계에서는 우선 **방어 코드 버전**부터 적용해도 충분합니다.

## 이슈 2. 프로필 소개 문구는 표시용 텍스트와 실제 값을 분리해야 합니다

### 왜 고쳐야 하나요

현재는 비어 있는 자기소개를 화면에 보여주기 위한 문구가 그대로 DOM에 들어가고, 수정 모달은 그 DOM 텍스트를 다시 읽어서 입력값으로 사용하고 있어요.

이러면 안내 문장이 실제 사용자 데이터처럼 섞여 들어갈 수 있어요.

### 목표

- "실제 bio 값"과 "비어 있을 때 보여줄 문구"를 분리하기
- 프로필 화면을 다시 그리는 코드를 한 함수로 모으기
- 문자열이 여러 곳에 흩어지지 않게 하기

### 추천 구조

`profile/profile.js`에 상수를 만들고, 렌더 함수를 분리하세요.

```js
const EMPTY_BIO_TEXT = '본인을 표현하는 글을 작성해보세요.';

function renderProfile(user) {
  document.getElementById('name').textContent = user.name;
  document.getElementById('role').textContent =
    user.role === 'ADMIN' ? '관리자' : '게스트';

  document.getElementById('bio').textContent = user.bio || EMPTY_BIO_TEXT;
}
```

그리고 수정 모달을 열 때는 DOM 문장을 읽지 말고, **현재 사용자 데이터**를 읽는 방식으로 바꾸는 것이 좋아요.

```js
let currentUser = null;

function renderProfile(user) {
  currentUser = user;
  document.getElementById('bio').textContent = user.bio || EMPTY_BIO_TEXT;
}

document.getElementById('editProfileBtn').addEventListener('click', () => {
  editBio.value = currentUser?.bio || '';
  openModal(editProfileModal);
});
```

이 구조의 장점:

- 안내 문장과 실제 값이 섞이지 않아요.
- fetch 후 렌더와 저장 후 렌더가 같은 함수를 쓰므로 문구가 달라질 일이 없어요.
- 나중에 `bio` 외에 다른 필드가 늘어나도 같은 패턴으로 확장할 수 있어요.

## 이슈 3. 지역 삭제는 "값을 안 보냄"이 아니라 "삭제하겠다고 명시해서 보냄"으로 처리해야 합니다

### 왜 고쳐야 하나요

사용자가 `선택 안함`을 골랐다는 것은 "지역을 비우고 싶다"는 뜻이에요. 그런데 현재 코드는 빈 문자열이면 `region` 자체를 요청 본문에 넣지 않아요.

그건 삭제 요청이 아니에요. 그냥 "아무 말도 안 했다"에 가까워요.

### 목표

- 사용자가 지역을 지우려는 의도를 서버에 명확하게 전달하기
- 화면의 지역 표시와 실제 데이터가 일치하게 유지하기

### 먼저 확인할 것

백엔드가 어떤 값을 "삭제"로 받아들이는지 먼저 확인해야 해요.

- `region: null`
- `region: ''`

둘 중 무엇을 쓰는지 API 규약이 있어야 해요.

### 예시 1: 빈 문자열을 삭제로 쓰는 경우

```js
const data = {
  bio: editBio.value.trim(),
  region: editRegion.value,
};
```

이렇게 하면 `선택 안함`일 때도 `region: ''`가 서버로 전달돼요.

### 예시 2: `null`을 삭제로 쓰는 경우

```js
const data = {
  bio: editBio.value.trim(),
  region: editRegion.value || null,
};
```

### 화면 갱신도 같이 고쳐야 합니다

지금은 숨길 때 `#region` 텍스트를 비우지 않아서 예전 값이 남을 수 있어요.

```js
if (user.region) {
  region.textContent = user.region;
  regionDot.classList.remove('hidden');
} else {
  region.textContent = '';
  regionDot.classList.add('hidden');
}
```

그리고 수정 모달을 열 때도 DOM에서 다시 읽지 말고 `currentUser.region`에서 읽는 것이 더 안전해요.

```js
editRegion.value = currentUser?.region || '';
```

### 핵심 정리

- 선택 안함 = 삭제 의도
- 삭제 의도는 요청 본문에 명시
- 숨긴다고 값이 지워지는 것은 아님

이 세 가지를 꼭 기억하면 비슷한 실수를 많이 줄일 수 있어요.

## 이슈 4. 예약 상세의 모달 단계는 `showCancelStep()` 같은 한 함수에서 관리하세요

### 왜 고쳐야 하나요

현재는 버튼마다 어떤 모달을 열고 닫아야 하는지 직접 써 놓았어요. 이 구조는 상태가 늘어날수록 빠르게 복잡해져요.

초보 단계에서는 "화면 단계"를 문자열 하나로 관리하면 훨씬 이해하기 쉬워요.

### 목표

- "지금은 어떤 단계인가?"를 한 함수가 책임지게 만들기
- 이벤트 핸들러는 "무슨 일이 일어났는지"만 말하게 만들기

### 추천 구조

```js
function showCancelStep(step) {
  closeModal(elements.cancelModal);
  closeModal(elements.cancelConfirmModal);
  closeModal(elements.cancelFaultModal);

  if (step === 'closed') {
    closeModal(elements.modalContainer);
    return;
  }

  openModal(elements.modalContainer);

  if (step === 'confirm') {
    openModal(elements.cancelModal);
  }

  if (step === 'success') {
    openModal(elements.cancelConfirmModal);
  }

  if (step === 'error') {
    openModal(elements.cancelFaultModal);
  }
}
```

이렇게 만들면 이벤트 핸들러는 훨씬 단순해져요.

```js
elements.cancel.addEventListener('click', () => {
  showCancelStep('confirm');
});

elements.cancelNo.addEventListener('click', () => {
  showCancelStep('closed');
});

elements.cancelYes.addEventListener('click', async () => {
  try {
    await removeReservation(reservationId);
    showCancelStep('success');
  } catch {
    elements.cancelFault.textContent =
      '예약을 취소하지 못했습니다.\n다시 시도해주세요.';
    showCancelStep('error');
  }
});
```

이 구조의 장점:

- 모달 상태 변경 규칙이 한곳에 모여요.
- 새 단계가 생겨도 `showCancelStep()`만 수정하면 돼요.
- 이벤트 핸들러는 읽기 쉬워져요.

이게 바로 응집도를 높이는 방법이에요.

## 마지막으로: 이번 수정에서 꼭 연습하면 좋은 습관

이번 리뷰와 해결 가이드에서 가장 중요한 습관은 아래 네 가지예요.

1. DOM에 보이는 문장을 데이터처럼 다시 읽지 않기
2. 숨김 처리와 값 비우기를 같은 것으로 생각하지 않기
3. 공통 유틸은 호출하는 쪽의 실수까지 방어하도록 만들기
4. 한 화면의 단계 전환은 한 함수로 모으기

이 네 가지를 익히면 HTML, TailwindCSS, JavaScript만으로도 코드 품질이 훨씬 좋아집니다.
