## 0. 먼저, 좋아진 점부터 볼게요

이번 반영분은 "조금씩 공통화하기"가 실제 코드에 보이기 시작했다는 점이 좋았어요.

- `src/components/modal.js`를 새로 만들면서 로그인, 회원가입, 예약 요청, 예약 상세, 프로필 페이지가 같은 방향으로 정리되기 시작했어요.
- `src/api/client.js`에서 `error.status`를 붙이기 시작한 것은 아주 좋은 습관이에요. 이제 문자열이 아니라 상태 코드로 분기할 수 있어서 코드가 더 안전해졌어요.
- `reservations-detail/index.js`가 자체 `getToken()` 구현을 지우고 `src/utils/auth.js`를 쓰게 된 점도 좋아요. 인증 로직은 한곳에 모일수록 유지보수가 쉬워져요.
- `profile` 화면은 이전보다 정보량이 많아지고, 수정 모달도 추가되어 사용자 경험이 분명히 좋아졌어요.

이번 리뷰의 핵심도 그대로예요.

> **응집도는 높게**: 관련 있는 일은 한곳에 모으기
> **결합도는 낮게**: 한곳을 바꿨을 때 다른 곳이 덜 흔들리게 만들기

이제부터는 "잘 돌아가지만, 실제 사용하면 문제될 수 있는 부분"을 중심으로 볼게요.

## 1. `modal.js`의 `openCount`는 현재 모달 상태를 정확히 표현하지 못합니다 (중요도: 높음)

**관련 파일:** `src/components/modal.js:1-18`, `reservations-detail/index.js:125-147`

지금 `modal.js`는 모달을 열 때마다 `openCount++`, 닫을 때마다 `openCount--`를 하고 있어요.

```js
let openCount = 0;

export function openModal(element) {
  if (!element) return;
  element.classList.remove('hidden');
  element.classList.add('flex');
  openCount++;
  document.body.classList.add('overflow-hidden');
}

export function closeModal(element) {
  if (!element) return;
  element.classList.remove('flex');
  element.classList.add('hidden');
  openCount = Math.max(0, openCount - 1);
  if (openCount === 0) {
    document.body.classList.remove('overflow-hidden');
  }
}
```

문제는 이 방식이 **"정말 열려 있던 모달을 닫은 것인지"** 확인하지 않는다는 점이에요.

`reservations-detail/index.js`를 보면 예약 취소 버튼을 눌렀을 때 이렇게 동작해요.

```js
openModal(elements.modalContainer);
openModal(elements.cancelModal);
closeModal(elements.cancelConfirmModal);
closeModal(elements.cancelFaultModal);
```

여기서 `cancelConfirmModal`과 `cancelFaultModal`은 처음부터 `hidden` 상태예요. 즉, **열리지도 않았는데 `closeModal()`이 호출되고 있어요.**

그 결과:

1. `modalContainer`를 열며 `openCount = 1`
2. `cancelModal`을 열며 `openCount = 2`
3. 이미 닫혀 있는 `cancelConfirmModal`을 "닫으며" `openCount = 1`
4. 이미 닫혀 있는 `cancelFaultModal`을 "닫으며" `openCount = 0`

이 시점에 화면에는 모달이 열려 있는데, `body`의 `overflow-hidden`은 풀려버려요. 즉, **배경 스크롤이 다시 가능해지는 상태 버그**가 생겨요.

이건 왜 중요한가요?

- 공통 컴포넌트는 여러 페이지가 믿고 쓰는 코드예요.
- 공통 컴포넌트가 내부 상태를 잘못 관리하면, 버그가 여러 페이지로 퍼져요.
- 즉, 각 페이지가 `modal.js`의 내부 규칙에 강하게 묶이게 되고, 결합도가 높아져요.

**핵심 문제:** `openCount`가 "실제 열린 모달 수"가 아니라 "함수 호출 횟수 차이"를 세고 있어요.

## 2. 프로필 소개 문구가 "화면 표시용 텍스트"에 결합되어 있어 다시 수정할 때 버그가 납니다 (중요도: 높음)

**관련 파일:** `profile/profile.js:38-39`, `profile/profile.js:95-99`, `profile/profile.js:116-117`

현재 프로필 소개(`bio`)는 비어 있을 때 화면에 기본 문구를 보여줘요.

```js
document.getElementById('bio').textContent =
  user.bio || '본인을 표현하는 글을 작성해보세요.';
```

그런데 저장 후에는 다른 문구를 써요.

```js
document.getElementById('bio').textContent =
  user.bio || '자기소개를 입력해주세요.';
```

그리고 수정 모달을 열 때는 첫 번째 문구만 검사해서 textarea를 비워요.

```js
const bio = document.getElementById('bio').textContent;
editBio.value = bio === '본인을 표현하는 글을 작성해보세요.' ? '' : bio;
```

이렇게 되면 어떤 일이 생길까요?

1. 사용자가 자기소개를 비운 채 저장함
2. 화면에는 `'자기소개를 입력해주세요.'`가 표시됨
3. 다시 수정 버튼을 누름
4. 위 조건문은 이 문구를 빈 값으로 보지 못함
5. textarea 안에 안내 문구가 실제 값처럼 들어감

즉, **안내 문장과 실제 사용자 데이터가 섞이는 버그**예요.

이건 초보 개발자분들이 특히 자주 만나게 되는 문제예요.

- "사용자에게 보여주는 문장"
- "서버에 저장할 실제 값"

이 둘은 반드시 분리해야 해요. 지금은 이 둘이 같은 DOM 텍스트 하나에 묶여 있어서 결합도가 높아졌어요.

추가로 같은 렌더링 로직이 `fetchProfile()`과 저장 성공 처리에 각각 따로 있어요. 그래서 기본 문구가 서로 달라졌어요. 이것도 응집도가 낮아졌다는 신호예요.

## 3. 지역 선택에서 `선택 안함`을 골라도 실제로는 지역이 삭제되지 않을 가능성이 큽니다 (중요도: 높음)

**관련 파일:** `profile/profile.js:108-124`

현재 저장 코드는 이렇게 되어 있어요.

```js
const data = { bio: editBio.value };
if (editRegion.value) data.region = editRegion.value;
```

문제는 사용자가 `선택 안함`을 고르면 `editRegion.value`가 빈 문자열이 되고, 그러면 `data.region`이 아예 들어가지 않는다는 거예요.

즉, 서버로는 이런 데이터가 가요.

```js
{ bio: '...' }
```

이 상태에서 서버는 보통 두 가지 중 하나로 동작해요.

- `region`이 없으니 기존 값을 유지한다.
- `region`이 없으니 아무 변경도 하지 않는다.

둘 중 어느 쪽이든, 사용자는 분명 "지역 삭제"를 눌렀는데 실제로는 삭제되지 않을 수 있어요.

게다가 화면 갱신도 완전히 안전하지 않아요.

```js
if (user.region) {
  document.getElementById('region').textContent = user.region;
  document.getElementById('regionDot').classList.remove('hidden');
} else {
  document.getElementById('regionDot').classList.add('hidden');
}
```

`else`에서는 `regionDot`만 숨기고 `#region`의 텍스트는 비우지 않아요. 그러면 이전 지역명이 DOM 안에 그대로 남을 수 있어요. 이후 다시 수정 모달을 열면:

```js
editRegion.value = document.getElementById('region').textContent || '';
```

숨겨진 예전 값이 다시 select에 들어갈 수 있어요.

즉, 이 코드는 현재:

- 화면 상태
- 폼 상태
- 서버로 보내는 값

세 가지가 서로 다를 수 있어요. 이것은 유지보수할수록 버그가 커지는 전형적인 신호예요.

## 4. 예약 상세의 모달 단계 전환 로직이 이벤트 핸들러마다 흩어져 있어 결합도가 높습니다 (중요도: 중간)

**관련 파일:** `reservations-detail/index.js:125-157`, `reservations-detail/index.html:29-74`

예약 상세 페이지는 사실상 하나의 흐름을 가지고 있어요.

1. 취소 확인 모달
2. 취소 성공 모달
3. 취소 실패 모달

그런데 지금은 이 흐름을 "하나의 상태"로 관리하지 않고, 각 버튼 이벤트에서 직접 DOM을 열고 닫고 있어요.

```js
elements.cancel.addEventListener('click', () => {
  openModal(elements.modalContainer);
  openModal(elements.cancelModal);
  closeModal(elements.cancelConfirmModal);
  closeModal(elements.cancelFaultModal);
});

elements.cancelYes.addEventListener('click', async () => {
  try {
    await removeReservation(reservationId);
    closeModal(elements.cancelModal);
    openModal(elements.cancelConfirmModal);
    closeModal(elements.cancelFaultModal);
  } catch (e) {
    closeModal(elements.cancelModal);
    closeModal(elements.cancelConfirmModal);
    openModal(elements.cancelFaultModal);
  }
});
```

이 구조의 문제는 각 핸들러가 모두 다음 사실을 알고 있어야 한다는 거예요.

- 어떤 모달이 있는지
- 어떤 모달이 현재 열려 있어야 하는지
- 다른 모달은 무엇을 닫아야 하는지
- 배경 컨테이너는 언제 켜고 꺼야 하는지

즉, "예약 취소 단계 전환"이라는 하나의 관심사가 한 함수에 모이지 못하고 여러 핸들러에 흩어져 있어요. 이러면 나중에 상태가 하나만 늘어나도 수정 지점이 여러 군데가 됩니다.

초보 개발자 기준으로는 이런 식으로 생각하면 쉬워요.

> "지금 화면이 어떤 단계인지"를 한 함수가 결정하게 만들자.

예를 들어 `showCancelStep('confirm')`, `showCancelStep('success')`, `showCancelStep('error')`처럼 만들면, 페이지의 응집도가 훨씬 높아져요.

## 5. 이번 리뷰의 핵심 정리

이번 반영분은 분명 좋아졌어요. 특히 공통 모듈로 옮기려는 시도가 보인다는 점은 아주 좋습니다. 다만 공통화는 "파일을 만들었다"에서 끝나면 안 되고, **상태를 안전하게 관리하는 구조**까지 같이 가야 해요.

이번에 꼭 기억했으면 하는 포인트는 네 가지예요.

1. 공통 컴포넌트는 함수 호출 횟수가 아니라 실제 상태를 기준으로 동작해야 해요.
2. 사용자에게 보여주는 안내 문장과 서버에 저장할 실제 값은 절대로 같은 것으로 취급하면 안 돼요.
3. 비우기/삭제 같은 동작은 "값을 안 보낸다"가 아니라 "삭제 의도를 명시해서 보낸다"로 설계해야 해요.
4. 한 화면의 단계 전환은 여러 이벤트에 흩뿌리지 말고 한 함수로 모아야 응집도가 올라가요.

구체적인 수정 방법은 [review-20260415_해결.md](./review-20260415_해결.md)에 단계별로 적어둘게요.
