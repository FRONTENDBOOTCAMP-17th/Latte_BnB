## 2. 잘한 점

이전 리뷰에서 지적한 7개 항목을 수정하면서 프로젝트가 정말 크게 발전했어요!

### 공통 컴포넌트 설계가 훌륭해요

- `src/components/header.js`와 `footer.js`로 공통 헤더/푸터를 **컴포넌트화**한 건 정말 좋은 판단이에요. 각 페이지마다 같은 HTML을 복사하지 않고, `buildHeader()`와 `buildFooter()` 함수 하나로 해결하고 있어요.

```js
// src/main.js — 이 두 줄로 모든 페이지에 헤더/푸터가 추가돼요!
document.body.prepend(buildHeader());
document.body.append(buildFooter());
```

### API 연동을 실제로 구현했어요

- 회원가입, 로그인, 프로필 조회, 로그아웃, 회원탈퇴까지 **실제 API 통신**을 구현한 건 대단해요! `fetch`와 `async/await`를 제대로 활용하고 있어요.
- 에러 처리(`try/catch`)도 빠짐없이 하고 있고, `localStorage`로 토큰 관리까지 하고 있어요.

```js
// login/index.js — fetch + async/await + 에러 처리 훌륭!
const res = await fetch('...auth/login', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(loginData),
});
```

### 토스트 컴포넌트가 전문가 수준이에요

- `src/components/toast.js`는 `message`, `success`, `warn` 3가지 타입을 지원하고, Popover API와 CSS 트랜지션을 활용한 애니메이션까지 구현했어요. 이건 초보자 수준을 훨씬 넘어서는 시도에요!
- `@starting-style` 같은 최신 CSS 기능도 활용했어요. CSS 트랜지션의 진입/퇴장 애니메이션을 잘 다루고 있어요.

### 관리자 로그인이 모범적이에요

- `admin/login/index.html`은 이 프로젝트에서 **유일하게 `<form>` 태그를 올바르게 사용**하고 있어요! `<fieldset>`, `<legend>`까지 시맨틱하게 작성했고, `e.preventDefault()`로 기본 제출 동작도 잘 제어하고 있어요.

```html
<form method="post" id="adminLoginForm">
  <fieldset class="flex flex-col gap-8">
    <legend>관리자 로그인</legend>
    ...
  </fieldset>
</form>
```

### 숙소 상세 페이지가 완성도 높아요

- `accommodations-detail`은 모달, 이미지 갤러리 스크롤, 가격 포맷팅까지 구현했어요. Tailwind CSS로 깔끔하게 스타일링했고, `openModal`/`closeModal` 헬퍼 함수로 모달 관리를 잘 정리했어요.
- 스크롤 버튼의 `data-target`과 `data-dir` 속성 활용도 아주 스마트해요!

### 유효성 검사를 꼼꼼하게 했어요

- 로그인과 회원가입 모두 정규식으로 입력값 유효성 검사를 구현했어요. 아이디 규칙(`^[a-z0-9_]{4,20}$`), 비밀번호 규칙(`8자 이상, 영문+숫자`) 등 실무에서도 사용하는 수준이에요.

### API URL 상수를 분리했어요

- `src/constants.js`로 API 기본 URL을 한 곳에서 관리하도록 만들었어요. `profile.js`와 `admin/login/login.js`에서 잘 활용하고 있어요.

## 3. 개선하면 좋을 점

### 3-1. 회원가입 전화번호 유효성 검사에 버그가 있어요 (중요도: 높음)

**현재 상태:**

```js
// signup/index.js 83번째 줄
if (!signupData.phone && !regPn.test(signupData.phone)) {
```

**왜 개선하면 좋을까?**

`&&`(AND)를 쓰면 첫 번째 조건이 `false`일 때 두 번째를 확인하지 않아요. 즉:
- `""` (빈 값): `!""`는 `true` → 두 번째 검사 실행 → 에러 메시지 표시 (정상)
- `"abc"` (잘못된 값): `!"abc"`는 `false` → 두 번째 검사 **건너뜀** → 에러 없이 통과! (버그!)
- `"01012345678"` (정상): `!"01012345678"`는 `false` → **통과** (이건 맞지만 우연히 맞는 거에요)

결과적으로 `"abc"` 같은 잘못된 전화번호가 유효성 검사를 통과해서 서버에 전송돼요.

**추천:** `&&`를 `||`(OR)로 바꾸세요:

```js
// 수정 후: 비어있거나 OR 형식이 안 맞으면 에러
if (!signupData.phone || !regPn.test(signupData.phone)) {
```

### 3-2. 로그인 성공 alert가 토큰 저장보다 먼저 실행돼요 (중요도: 높음)

**현재 상태:**

```js
// login/index.js 71~81번째 줄
if (!res.ok) {
  throw new Error('HTTP 오류: ' + res.status);
} else {
  alert(`로그인 되었습니다.`);   // ① alert 먼저!
}

const data = await res.json();          // ② 그 다음 JSON 파싱
const personalData = data.data;
localStorage.setItem('accessToken', personalData.accessToken); // ③ 토큰 저장
location.href = `../`;                  // ④ 페이지 이동
```

**왜 개선하면 좋을까?**

`alert`가 먼저 뜨고 나서 JSON 파싱과 토큰 저장이 실행돼요. 만약 JSON 파싱에서 에러가 나면, 사용자는 "로그인 되었습니다"를 봤는데 실제로는 로그인이 안 된 상황이 될 수 있어요.

**추천:** 모든 처리가 끝난 후에 알림을 띄우세요:

```js
const data = await res.json();
const personalData = data.data;
localStorage.setItem('accessToken', personalData.accessToken);
alert(`로그인 되었습니다.`);  // 모든 처리 완료 후!
location.href = `../`;
```

### 3-3. toast.js에서 `buildMessage` 함수가 자기 자신을 호출해요 (중요도: 높음)

**현재 상태:**

```js
// src/components/toast.js 66~76번째 줄
function buildMessage(message, details = '') {
  // ...
  toastMessageNode = document.querySelector('#toastMessage');
  if (toastMessageNode === null) {
    toastMessageNode = buildMessage();    // ← 자기 자신을 호출! 무한 반복!
    toastNode.appendChild(toastMessageNode);
  }

  toastDetailNode = document.querySelector('#toastDetail');
  if (toastDetailNode === null && details) {
    toastDetailNode = buildDetail();      // ← 이 함수는 존재하지 않아요!
    toastNode.appendChild(toastDetailNode);
  }
```

**왜 개선하면 좋을까?**

- 68번째 줄: `buildMessage()`는 현재 함수 자체에요. `#toastMessage` DOM이 없으면 **무한 재귀 호출**로 브라우저가 멈춰요.
- 74번째 줄: `buildDetail()`은 정의되지 않은 함수라 에러가 나요. 올바른 이름은 `buildDetailNode()`에요.

**추천:**

```js
// 수정
toastMessageNode = buildMessageNode();   // buildMessage → buildMessageNode
toastDetailNode = buildDetailNode();     // buildDetail → buildDetailNode
```

### 3-5. login/signup 페이지에서 API URL을 하드코딩하고 있어요 (중요도: 중간)

**현재 상태:**

`src/constants.js`에 API URL을 잘 정의해두었는데, `login/index.js`와 `signup/index.js`에서는 사용하지 않고 URL을 직접 적고 있어요:

```js
// login/index.js 61번째 줄 — URL 직접 입력
const res = await fetch('https://api.fullstackfamily.com/api/lattebnb/v1/auth/login', ...);

// signup/index.js 91번째 줄 — 여기도 직접 입력
const res = await fetch('https://api.fullstackfamily.com/api/lattebnb/v1/auth/signup', ...);
```

반면 `admin/login/login.js`와 `profile/profile.js`는 `constants.js`를 잘 사용하고 있어요:

```js
// profile/profile.js — 올바른 방법!
import constants from '../src/constants.js';
const API_BASE = constants.API_BASE_URL;
const res = await fetch(`${API_BASE}/me/profile`, ...);
```

**왜 개선하면 좋을까?**

나중에 API 서버 주소가 바뀌면 `constants.js`만 수정하면 되는데, 하드코딩된 곳은 직접 찾아서 바꿔야 해요. 빠뜨리면 일부 페이지만 동작하는 버그가 생겨요.

**추천:** `login/index.js`와 `signup/index.js`에도 `constants.js`를 import해서 사용하세요.

### 3-7. login/signup에 `<form>` 태그가 없어요 (중요도: 중간)

**현재 상태:**

`login/index.html`과 `signup/index.html`은 입력 필드와 버튼이 `<div>` 안에 있고 `<form>`이 없어요. `admin/login/index.html`은 `<form>`을 잘 사용하고 있어서 차이가 나요.

**왜 개선하면 좋을까?**

`<form>` 없이 `<button type="submit">`을 쓰면:
- 기본 submit 동작이 없어서 JavaScript에만 의존해야 해요
- 브라우저 자동완성/비밀번호 저장 기능이 제한돼요
- 스크린 리더가 "이것은 입력 양식이다"를 인식하지 못해요

현재 Enter 키 동작을 각 input마다 일일이 `keydown` 이벤트로 구현하고 있는데, `<form>`을 쓰면 Enter로 자동 제출돼서 그 코드가 필요 없어져요.

**추천:** `admin/login/index.html`처럼 `<form>` + `e.preventDefault()` 패턴을 사용하세요.

### 3-8. accommodations-detail.js에서 API 데이터를 innerHTML에 직접 넣고 있어요 (중요도: 중간)

**현재 상태:**

```js
// accommodations-detail/accommodations-detail.js 34~39번째 줄
figure.innerHTML =
  `<img src="${img.url}" alt="${img.title || '숙소 이미지'}" .../>` +
  `<span ...>${img.title || ''}</span>` +
  `<span ...>${img.description || ''}</span>`;
```

**왜 개선하면 좋을까?**

`innerHTML`에 외부 데이터(API 응답)를 직접 넣으면, 만약 `img.title`에 `<script>alert('해킹!')</script>` 같은 코드가 들어있으면 그대로 실행될 수 있어요. 이걸 **XSS(크로스 사이트 스크립팅)** 공격이라고 해요.

지금은 직접 관리하는 API라 괜찮지만, 안전한 습관을 기르는 게 중요해요.

**추천:** `textContent`를 사용하면 HTML이 자동으로 이스케이프돼요:

```js
const span = document.createElement('span');
span.textContent = img.title || '';  // <script> 태그가 들어와도 안전!
```

또는 기본 HTML 구조는 `innerHTML`로 만들되, 데이터는 `textContent`로 넣으세요:

```js
figure.innerHTML = `<img class="..." /><figcaption><span class="title"></span><span class="desc"></span></figcaption>`;
figure.querySelector('img').src = img.url;
figure.querySelector('img').alt = img.title || '숙소 이미지';
figure.querySelector('.title').textContent = img.title || '';
figure.querySelector('.desc').textContent = img.description || '';
```

### 3-9. login/signup에서 에러 메시지에 `style.color = 'red'` 인라인 스타일을 쓰고 있어요 (중요도: 낮음)

**현재 상태:**

```js
// login/index.js 49번째 줄
result1.textContent = '아이디를 입력해주세요.';
result1.style.color = 'red';

// signup/index.js도 동일
```

HTML에 이미 Tailwind 클래스 `text-negative-500`이 적용되어 있어요:

```html
<p id="result1" class="... text-negative-500"></p>
```

**왜 개선하면 좋을까?**

Tailwind에 이미 빨간색이 설정되어 있으니 `style.color = 'red'`는 불필요해요. 인라인 스타일은 Tailwind 클래스보다 우선순위가 높아서 테마 색상과 달라질 수 있어요.

**추천:** `style.color = 'red'` 줄을 삭제하세요. Tailwind의 `text-negative-500`이 이미 에러 색상을 제공해요.

### 3-10. profile.js에서 토큰 없을 때 코드가 계속 실행돼요 (중요도: 낮음)

**현재 상태:**

```js
// profile/profile.js 10~14번째 줄
const token = localStorage.getItem('accessToken');
if (!token) {
  alert('로그인을 먼저 해주세요');
  location.href = '/login/';
}
// ← 여기서부터 아래 코드가 계속 실행돼요!
fetchProfile();  // 토큰 없이 API 호출 시도
```

**왜 개선하면 좋을까?**

`location.href`를 설정해도 JavaScript 실행이 즉시 멈추지 않아요. 페이지가 이동하기 전에 아래 코드가 실행되면서, 토큰 없이 API를 호출하게 돼요.

**추천:** 토큰 없을 때 나머지 코드가 실행되지 않도록 전체를 감싸세요:

```js
if (!token) {
  alert('로그인을 먼저 해주세요');
  location.href = '/login/';
} else {
  fetchProfile();
  // ... 나머지 초기화 코드
}
```
