## 2. 잘한 점

5일 만에 **7개 페이지와 6개 공통 컴포넌트**를 새로 추가한 건 정말 놀라운 속도에요!

### 이전 리뷰 항목을 꼼꼼하게 반영했어요

- 이전 리뷰에서 지적한 11개 항목 중 **7개를 완벽하게 수정**했어요. 특히 `toast.js`의 재귀 호출 버그 수정, `vite.config.js`에 12개 페이지 등록, `constants.js` 활용 등 핵심 항목들을 모두 해결했어요.

### 컴포넌트 설계가 점점 전문적이에요

- `RoomCard` 클래스는 **프라이빗 필드(`#`)와 캡슐화**를 활용해서 객체지향 프로그래밍의 모범 사례를 보여주고 있어요:

```js
// src/RoomCard.js — 프라이빗 필드로 데이터 보호!
#id; #title; #location; #thumbnailUrl; #pricing; #maxGuest;
#isWish = false; #element = null;
```

- `PreviewImage` 클래스도 같은 패턴으로 잘 작성했어요. 학생 수준에서 이렇게 `#` 프라이빗 필드를 활용하는 건 매우 인상적이에요!

### 달력 컴포넌트가 전문가 수준이에요

- `calendar.js`는 이 프로젝트에서 가장 복잡하고 완성도 높은 컴포넌트에요:
  - 체크인/체크아웃 날짜 선택
  - 예약 불가 날짜 블럭 처리
  - 날짜 범위 하이라이트
  - 달력 네비게이션 (이전/다음 월)
  - `setCalendarChange` 콜백으로 외부와 통신

```js
// calendar.js — 콜백 패턴으로 느슨한 결합 달성!
function setCalendarChange(call) {
  onCalendarChange = call;
}
```

이 콜백 패턴은 **결합도를 낮추는 좋은 방법**이에요. 달력은 "날짜가 바뀌었다"만 알려주고, 무엇을 할지는 호출하는 쪽이 결정해요.

### 관리자 페이지의 `<form>` + `<fieldset>` 활용이 모범적이에요

- `admin/add/index.html`은 `<form>`, `<fieldset>`, `<legend>`, `<label>`을 HTML 표준에 맞게 완벽하게 사용하고 있어요. 특히 `form="addForm"` 속성으로 외부 버튼에서 폼을 제출하는 고급 기법도 활용했어요:

```html
<!-- admin/add/index.html — form 속성으로 외부 버튼 연결! -->
<button type="submit" form="addForm" class="...">등록하기</button>
```

### API 연동 코드가 체계적이에요

- `adminLanding.js`에서 `fetchAccommodationList`와 `deleteAccommodation`을 **별도 모듈로 분리**한 건 정말 잘한 판단이에요. API 관련 코드와 UI 코드를 분리하면 나중에 API가 바뀌어도 한 곳만 수정하면 돼요.

```js
// admin/adminLanding.js — API 로직을 별도 모듈로 분리!
export async function fetchAccommodationList({ page, query, pageLimit = 20 } = {}) { ... }
export async function deleteAccommodation(id) { ... }
```

### 반응형 디자인이 일관적이에요

- 거의 모든 페이지에서 `sm:`, `md:`, `lg:`, `2xl:` 브레이크포인트를 활용한 반응형 디자인을 잘 적용하고 있어요. 특히 `index.html`의 숙소 그리드가 뷰포트에 따라 1~4열로 변하는 건 실무급이에요:

```html

```

### 이벤트 위임을 잘 활용하고 있어요

- `landing.js`와 `admin/index.js`에서 `document.addEventListener('click', ...)` 이벤트 위임 패턴을 사용해서, 동적으로 생성되는 버튼에도 이벤트가 잘 동작해요. 이건 초보 개발자가 놓치기 쉬운 부분인데 잘 처리했어요!

## 3. 개선하면 좋을 점

### 3-1. 인증 체크 코드가 4곳에서 거의 동일하게 복사되어 있어요 (중요도: 높음)

**현재 상태:**

거의 같은 인증 체크 코드가 4곳에 복사-붙여넣기 되어 있어요:

```js
// wishlist/index.js 7~29줄
const token = localStorage.getItem('accessToken');
if (token) {
  const res = await fetch(`${constants.API_BASE_URL}/me/profile`, {
    method: 'GET',
    headers: { Authorization: `Bearer ${token}` },
  });
  if (!res.ok) {
    localStorage.removeItem('accessToken');
    location.replace('/login/');
  }
  const { success } = await res.json();
  if (success) {
    console.log('유효한 토큰입니다.');
    // ... UI 표시
  }
} else {
  location.replace('/login/');
}
```

```js
// admin/index.js 19~51줄 — 거의 동일한 코드!
// admin/add/index.js 20~40줄 — 또 동일한 코드!
// src/main.js 42~72줄 — checkAuth 함수 (약간 다른 형태)
```

**왜 개선하면 좋을까?**

이것은 **"응집도는 높이고, 결합도는 낮추자"** 원칙의 대표적인 위반 사례에요.

- **응집도(Cohesion)**: "관련 있는 기능은 한 곳에 모아두자". 인증 체크라는 하나의 기능이 4곳에 흩어져 있으면 응집도가 낮아요.
- **결합도(Coupling)**: 각 페이지가 인증 체크의 구체적인 구현(fetch URL, 헤더 형식, 에러 처리)에 직접 의존하면 결합도가 높아요. API 응답 형식이 바뀌면 4곳을 모두 수정해야 해요.

**추천:** `src/auth.js` 유틸리티를 만들어 한 곳에서 관리하세요:

```js
// src/auth.js (신규)
import constants from './constants.js';

export async function checkAuth({ redirectTo = '/login/' } = {}) {
  const token = localStorage.getItem('accessToken');
  if (!token) {
    location.replace(redirectTo);
    return null;
  }

  const res = await fetch(`${constants.API_BASE_URL}/me/profile`, {
    method: 'GET',
    headers: { Authorization: `Bearer ${token}` },
  });

  if (!res.ok) {
    localStorage.removeItem('accessToken');
    location.replace(redirectTo);
    return null;
  }

  const { data } = await res.json();
  return data;
}
```

```js
// wishlist/index.js — 인증 체크가 한 줄로!
import { checkAuth } from '../src/auth.js';

const user = await checkAuth();
if (!user) throw new Error('인증 필요');
// ... 나머지 로직
```

**왜 개선하면 좋을까?**

**추천:** `src/components/emptyState.js`를 만드세요:

```js
// src/components/emptyState.js (신규)
export function buildEmptyState(message) {
  const div = document.createElement('div');
  div.className = 'w-full min-h-140 col-span-3 flex gap-20 items-center justify-center text-lg font-semibold text-shark-600';

  const text = document.createTextNode(message + ' ');
  const highlight = document.createElement('span');
  highlight.className = 'inline-block text-[10rem] font-extrabold text-primary-500 align-text-top';
  highlight.textContent = '텅...';
  const suffix = document.createTextNode(' 비어있습니다.');

  div.append(text, highlight, suffix);
  return div;
}
```

```js
// wishlist/index.js — 한 줄로!
import { buildEmptyState } from '../src/components/emptyState.js';
roomList.appendChild(buildEmptyState('위시리스트가'));

// reservations-check/index.js — 메시지만 바꿔서 재사용!
result.appendChild(buildEmptyState('예약 목록이'));
```

`reservations-check/index.js`도 66~70줄에서 같은 패턴이에요:

```js
resDiv.innerHTML = `
  <img class="..." src="${amd.thumbnailUrl}" alt="${amd.title}">
  <p class="...">${amd.title}</p>
  <p class="...">${inmonth}월 ${inday}일 ~ ...</p>
`;
```

**왜 개선하면 좋을까?**

이전 3차 리뷰 3-8번에서 `accommodations-detail.js`의 같은 문제를 지적했는데, **새로 만든 페이지에서도 동일한 패턴**이 반복되고 있어요.

서버에서 받은 데이터(`title`, `location`, `thumbnailUrl`)를 `innerHTML`에 직접 넣으면 **XSS(크로스 사이트 스크립팅)** 위험이 있어요. 예를 들어 숙소 이름에 `<img src=x onerror=alert('해킹!')>`이 들어있으면 그대로 실행돼요.

**추천:** HTML 구조는 `innerHTML`로 만들되, 데이터는 `textContent`와 속성으로 삽입하세요:

```js
// reservations-check/index.js — 안전한 방법
resDiv.innerHTML = `
  <img class="w-full h-40 object-cover rounded-md" />
  <p class="mt-2 font-bold"></p>
  <p class="text-sm text-gray-500"></p>
`;
// 데이터는 안전하게!
resDiv.querySelector('img').src = amd.thumbnailUrl;
resDiv.querySelector('img').alt = amd.title;
resDiv.querySelector('.font-bold').textContent = amd.title;
resDiv.querySelector('.text-gray-500').textContent = `${inmonth}월 ${inday}일 ~ ${outmonth}월 ${outday}일 | ${sch.nights}박`;
```

### 3-4. admin/index.html에 `<style>` 태그로 CSS를 직접 작성하고 있어요 (중요도: 중간)

**현재 상태:**

<!-- admin/index.html 9~15줄 -->

    grid-column: 1 / -1;
    margin-top: 1.25rem;
  }
  #pagination {
    margin-bottom: 1.25rem;
  }

**왜 개선하면 좋을까?**

이 프로젝트는 **Tailwind CSS를 사용하기로 했기 때문에**, CSS는 최소한으로만 사용하고 가능한 한 Tailwind 유틸리티 클래스로 작성해야 해요. `<style>` 태그로 CSS를 직접 쓰면:
- Tailwind 클래스와 섞여서 어디서 스타일이 적용되는지 찾기 어려워요
- 다른 팀원이 Tailwind 클래스만 보고 스타일을 이해할 수 없어요
- 스타일이 분산되어 응집도가 낮아져요

**추천:** Tailwind 클래스로 대체하세요. `grid-column: 1 / -1`은 Tailwind의 `col-span-full`이에요:

```html
<!-- style 태그 삭제하고, JS에서 동적 생성 시 클래스 추가 -->
<!-- admin/index.js의 adminLogo.build()와 pagination에 클래스를 추가 -->
```

`adminLogo.js`에서 이미 `justify-self-center`를 쓰고 있으니, `col-span-full mt-5`를 추가하면 돼요:

```js
// adminLogo.js — className에 추가
adminLogo.className = 'col-span-full mt-5 w-fit h-15 text-primary-300 ...';
```

`pagination.js`에서도 빌드 시 `col-span-full mt-5 mb-5`를 추가하면 `<style>` 태그를 완전히 제거할 수 있어요.

### 3-5. 모달 열기/닫기 방식이 파일마다 달라요 (중요도: 중간)

**현재 상태:**

프로젝트에서 모달을 열고 닫는 방법이 **3가지**나 섞여 있어요:

**방법 1: CSS 클래스 토글** (accommodations-detail, profile)
```js
// accommodations-detail/accommodations-detail.js 86~93줄
function openModal(modal) {
  modal.classList.add('active');
  document.body.style.overflow = 'hidden';
}
function closeModal(modal) {
  modal.classList.remove('active');
  document.body.style.overflow = '';
}
```

**방법 2: style.display 직접 변경** (reservation-request)
```js
// reservation-request/reservation-request.js 199~201줄
calModal.style.display = 'flex';
document.body.style.overflow = 'hidden';
```

**방법 3: classList 토글 hidden** (admin/add)
```js
// admin/add/index.js 91줄
modal.classList.add('hidden');
```

**왜 개선하면 좋을까?**

같은 "모달 열기/닫기" 기능인데 방식이 3가지면, 팀원이 혼란스러워요. 또 Tailwind를 쓰는 프로젝트에서 `style.display = 'flex'`는 인라인 스타일이라 Tailwind와 충돌할 수 있어요.

**추천:** `accommodations-detail`의 **CSS 클래스 토글 방식**이 가장 깔끔해요. 이걸 표준 패턴으로 통일하세요:

```js
// 모든 모달에서 통일
function openModal(modal) {
  modal.classList.add('active');
  document.body.style.overflow = 'hidden';
}
function closeModal(modal) {
  modal.classList.remove('active');
  document.body.style.overflow = '';
}
```

HTML에서 모달 기본 스타일에 `.hidden`이 아닌 `opacity-0 invisible` + `[&.active]:opacity-100 [&.active]:visible`을 사용하세요 (accommodations-detail 패턴).

reservation-request의 모달 HTML에서 `hidden` 대신 이 패턴으로 바꾸면 트랜지션 애니메이션도 적용할 수 있어요.

### 3-7. accommodations-detail에서 onclick 인라인 이벤트 핸들러를 쓰고 있어요 (중요도: 중간)

**현재 상태:**

```html
<!-- accommodations-detail/index.html 14줄 -->
<button class="..." onclick="location.href = '/'">
```

**왜 개선하면 좋을까?**

이 프로젝트의 다른 모든 곳에서는 `addEventListener`를 사용해서 이벤트를 처리하고 있는데, 이 한 곳만 HTML에 `onclick`을 직접 썼어요.

- `onclick` 인라인 핸들러는 HTML과 JavaScript가 섞이게 되어 **관심사 분리** 원칙에 어긋나요
- 다른 곳에서 사용하는 `addEventListener` 방식과 일관성이 깨져요
- CSP(Content Security Policy)를 적용하면 인라인 핸들러가 차단될 수 있어요

**추천:** 뒤로가기 버튼에 id를 추가하고 JS에서 이벤트를 등록하세요:

```html
<!-- HTML -->
<button id="btn-back" class="...">
```

```js
// accommodations-detail.js
document.getElementById('btn-back').addEventListener('click', () => {
  location.href = '/';
});
```

특히 자랑스러운 점:
- **`RoomCard`와 `PreviewImage`의 프라이빗 필드 활용**은 JavaScript 클래스의 모범 사례에요
- **달력 컴포넌트**는 예약 불가 날짜 블럭, 범위 선택, 콜백 패턴까지 구현한 전문가급 코드에요

- **`adminLanding.js`의 API 분리**는 "응집도는 높이고 결합도는 낮추자"를 잘 실천한 사례에요

이번 리뷰의 핵심 메시지는 **"반복되는 코드를 공통 모듈로 분리하자"**에요. 인증 체크, 빈 화면 UI, 모달 패턴처럼 여러 곳에서 복사-붙여넣기 하는 코드를 하나의 모듈로 만들면, 코드 양도 줄고 유지보수도 훨씬 쉬워져요. 이것이 바로 **응집도를 높이고 결합도를 낮추는** 핵심 방법이에요!
