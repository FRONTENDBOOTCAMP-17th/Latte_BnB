## 2. 잘한 점

프로젝트 시작 단계에서 이미 많은 준비를 잘 해두었어요! 하나씩 살펴볼게요.

### 개발 환경 설정이 훌륭해요

- **Vite + Tailwind CSS v4** 조합을 사용하고 있어요. 이건 2026년 기준 가장 현대적이고 빠른 개발 환경이에요. `@tailwindcss/vite` 플러그인을 사용해서 Tailwind CSS v4의 공식 방법으로 잘 연동했어요.

```js
// vite.config.js
import tailwindcss from '@tailwindcss/vite';
export default defineConfig({
  plugins: [tailwindcss()],
});
```

- Tailwind CSS v4의 새로운 `@theme` 문법을 사용해서 커스텀 색상을 정의한 것도 아주 좋아요. 최신 문법을 잘 공부하고 적용했어요!

### ESLint 설정이 꼼꼼해요

- `eslint.config.mjs`에서 아주 좋은 규칙들을 설정해 두었어요:
  - `eqeqeq: 'error'` → `===` 사용 강제 (버그 예방에 아주 중요해요!)
  - `no-var: 'error'` → `var` 대신 `const`/`let` 사용
  - `prefer-const: 'error'` → 값이 바뀌지 않으면 `const` 사용
  - `no-implicit-globals: 'error'` → 전역 변수 생성 금지

이런 규칙들은 코드 품질을 유지하는 데 정말 큰 도움이 돼요. 팀 프로젝트에서 코딩 규칙을 미리 정해둔 건 정말 프로답습니다!

### Prettier 설정도 잘 되어 있어요

- 세미콜론, 따옴표, 들여쓰기 등 코드 스타일을 `.prettierrc`에 통일해 두었어요. 팀원 3명이 같은 스타일로 코드를 작성할 수 있게 해주는 중요한 설정이에요.
- `.prettierignore`에 `*.md`, `package.json`, `package-lock.json`을 제외한 것도 적절해요.

### HTML 기본 구조가 정확해요

- `<!doctype html>`, `lang="ko-KR"`, `charset="UTF-8"`, `viewport` 메타태그가 모두 올바르게 설정되어 있어요. 이건 웹 접근성과 반응형 디자인의 기초인데, 처음부터 잘 잡아놓았어요!

### 문서화에 많은 노력을 기울였어요

- API 명세서, 프로토타입 디자인 시안, 구현 과정 문서 등을 꼼꼼하게 준비했어요.
- PR 템플릿(`PULL_REQUEST_TEMPLATE.MD`)도 만들어두었고, 이는 팀 협업의 품질을 높여주는 좋은 습관이에요.
- 9개 페이지에 대한 디자인 시안 문서를 모두 작성해 둔 것은 기획 단계를 충실히 거쳤다는 증거에요. 대단해요!

### 커스텀 테마 색상 팔레트

- `src/style.css`에 primary, secondary, tertiary, shark(그레이스케일), positive, negative 총 6가지 색상 팔레트를 50~950 단계로 세밀하게 정의했어요. 디자인 시스템을 고려한 접근이에요!

## 3. 개선하면 좋을 점

### 3-1. primary와 secondary 색상 팔레트가 거의 동일해요 (중요도: 높음)

**현재 상태:**

`src/style.css`에서 primary와 secondary 색상이 거의 같은 값으로 정의되어 있어요:

```css
/* primary */
--color-primary-400: #ff8194;
--color-primary-700: #c30d3b;

/* secondary */
--color-secondary-400: #ff6d83;
--color-secondary-700: #c30d3a;
```

11개 단계(50~950) 중 400번과 700번만 아주 미세하게 다르고 나머지 9개는 완전히 같아요.

**왜 개선하면 좋을까?**

primary(주 색상)와 secondary(보조 색상)는 서로 다른 역할을 하기 위해 존재해요. 같은 색이면 두 개를 나눌 이유가 없어요. 예를 들어 에어비앤비에서는 주 색상(핑크/빨강)과 보조 색상(진한 회색이나 다른 톤)을 구분해서 사용해요.

**추천:** Figma 디자인 시안에서 사용하는 색상을 확인하고, secondary 팔레트를 primary와 구별되는 다른 색상으로 변경하세요. 만약 의도적으로 같은 계열이라면, tertiary 팔레트와 역할을 정리해서 팀 내에서 "어떤 상황에 어떤 색을 쓸지" 합의하는 게 좋아요.

### 3-2. README의 파일 링크에 `.md`가 중복되어 있어요 (중요도: 높음)

**현재 상태:**

`README.md` 49~53번째 줄에서 일부 링크에 `.md`가 두 번 붙어 있어요:

```html
<li><a href="./docs/prototype-design-draft/accommodation_detail.md.md">숙소 상세 페이지</a></li>
<li><a href="./docs/prototype-design-draft/reservation_request.md.md">예약 신청 페이지</a></li>
<li><a href="./docs/prototype-design-draft/reservation_detail.md.md">예약 상세 페이지</a></li>
<li><a href="./docs/prototype-design-draft/my_reservation.md.md">내 예약 목록 페이지</a></li>
```

**왜 개선하면 좋을까?**

실제 파일 이름은 `accommodation_detail.md`인데 링크는 `accommodation_detail.md.md`로 되어 있어서, GitHub에서 링크를 클릭하면 "파일을 찾을 수 없음" 오류가 나와요.

**추천:** `.md.md`를 `.md`로 수정하세요.

```html
<!-- 수정 후 -->
<li><a href="./docs/prototype-design-draft/accommodation_detail.md">숙소 상세 페이지</a></li>
<li><a href="./docs/prototype-design-draft/reservation_request.md">예약 신청 페이지</a></li>
<li><a href="./docs/prototype-design-draft/reservation_detail.md">예약 상세 페이지</a></li>
<li><a href="./docs/prototype-design-draft/my_reservation.md">내 예약 목록 페이지</a></li>
```

### 3-3. README의 프로젝트 구조가 실제 구조와 달라요 (중요도: 중간)

**현재 상태:**

README에 적힌 프로젝트 구조:
```
latte-bnb/
├── css/
│   ├── theme.css
│   ├── common.css
│   ├── main.css
├── js/
│   ├── common.js
│   ├── main.js
├── assets/
│   ├── images/
```

실제 프로젝트 구조:
```
Latte_BnB/
├── src/
│   ├── main.js
│   └── style.css
```

**왜 개선하면 좋을까?**

README의 구조와 실제 코드가 다르면, 새로 합류하는 팀원이나 코드를 읽는 사람이 혼란스러워질 수 있어요. 특히 Vite를 사용하고 있으니, Vite의 일반적인 구조(`src/` 폴더 기반)를 기준으로 README를 업데이트하는 게 좋아요.

**추천:** 실제 개발을 더 진행한 후, 실제 파일 구조에 맞춰서 README를 업데이트하세요. 또는 지금 시점에서 팀이 사용할 폴더 구조를 확정하고 README에 반영하세요.

### 3-4. Vite 멀티 페이지 설정이 필요해요 (중요도: 중간)

**현재 상태:**

```js
// vite.config.js
export default defineConfig({
  plugins: [tailwindcss()],
});
```

현재 `vite.config.js`에는 Tailwind 플러그인만 설정되어 있고, 멀티 페이지 설정이 없어요.

**왜 개선하면 좋을까?**

README에 따르면 메인 페이지, 숙소 상세, 위시리스트, 로그인 등 여러 HTML 페이지를 만들 계획이에요. Vite는 기본적으로 `index.html` 하나만 빌드하기 때문에, 여러 HTML 파일을 사용하려면 `build.rollupOptions.input`에 각 페이지를 등록해야 해요.

**추천:** 새로운 HTML 페이지를 추가할 때 `vite.config.js`에 멀티 페이지 설정을 추가하세요:

```js
import { resolve } from 'path';
import { defineConfig } from 'vite';
import tailwindcss from '@tailwindcss/vite';

export default defineConfig({
  plugins: [tailwindcss()],
  build: {
    rollupOptions: {
      input: {
        main: resolve(__dirname, 'index.html'),
        detail: resolve(__dirname, 'detail.html'),
        wishlist: resolve(__dirname, 'wishlist.html'),
        login: resolve(__dirname, 'login.html'),
        // 페이지가 추가되면 여기에 등록
      },
    },
  },
});
```

### 3-5. `<title>` 태그를 더 설명적으로 작성하면 좋아요 (중요도: 낮음)

**현재 상태:**

```html
<title>latte-bnb</title>
```

**왜 개선하면 좋을까?**

`<title>` 태그는 브라우저 탭에 표시되는 텍스트에요. "latte-bnb"만으로는 이 사이트가 어떤 서비스인지 사용자가 바로 알기 어려워요. 검색엔진(SEO)에서도 `<title>`은 중요한 요소에요.

**추천:**

```html
<title>라떼비앤비 - 특별한 숙소를 찾아보세요</title>
```

### 3-6. `index.html`의 `<body>`에 기본 콘텐츠가 없어요 (중요도: 낮음)

**현재 상태:**

```html
<body>
  <script type="module" src="/src/main.js"></script>
</body>
```

**왜 개선하면 좋을까?**

현재 `<body>`에는 스크립트 태그만 있고 아무런 HTML 콘텐츠가 없어요. 프로젝트 시작 단계이니 당연하지만, 페이지를 만들 때 시맨틱 태그(`<header>`, `<main>`, `<footer>` 등)로 큰 뼈대를 먼저 잡고 시작하면 작업이 훨씬 수월해져요.

**추천:** 가장 먼저 구현할 메인 페이지의 기본 뼈대를 시맨틱 태그로 작성해보세요:

```html
<body>
  <header>
    <!-- 네비게이션 바 -->
  </header>
  <main>
    <!-- 메인 콘텐츠 -->
  </main>
  <footer>
    <!-- 푸터 -->
  </footer>
  <script type="module" src="/src/main.js"></script>
</body>
```

### 3-7. favicon(파비콘)이 설정되어 있지 않아요 (중요도: 낮음)

**현재 상태:**

`index.html`의 `<head>`에 favicon 링크가 없어요.

**왜 개선하면 좋을까?**

파비콘(favicon)은 브라우저 탭의 작은 아이콘이에요. 설정하지 않으면 브라우저가 기본 아이콘을 보여주거나, 개발자 도구에 404 에러가 나타나요. 작은 부분이지만 사이트의 완성도를 높여주는 요소에요.

**추천:** 프로젝트 로고나 간단한 아이콘을 준비해서 `<head>`에 추가하세요:

```html
<link rel="icon" type="image/svg+xml" href="/favicon.svg" />
```

## 5. 다음 단계 제안

1. **README 링크 오류 수정** — 가장 빠르게 고칠 수 있고, GitHub에서 바로 확인 가능해요

5. **팀 내 폴더 구조 합의** — `src/` 하위에 `pages/`, `components/`, `assets/` 등을 어떻게 나눌지 팀원들과 논의하세요
